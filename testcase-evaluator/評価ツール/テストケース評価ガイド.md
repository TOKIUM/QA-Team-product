## 1. 概要

### 何ができるか
- テスト項目書のExcelファイルを読み込み、運用ルール違反を自動検出
- 100点満点のスコアを算出
- 違反箇所を一覧表示し、修正推奨事項を提示

### 評価基準の元になるもの
- **運用ルール**（`改善要望/運用ルール.png`）
  - 期待値：曖昧に書かない / 1ケースにつき期待値は一つ
  - 全体の表現：我流の単語は使わない
  - 手順：極力省かない

### 自動チェックで検出できること / できないこと

| 自動チェック可能 | 手動チェックが必要 |
|---|---|
| 曖昧な期待値（NGワードパターンマッチ） | 期待値の具体性が十分かの判断 |
| 1セルに「こと」が2回以上（複数期待値の混在） | 「こと」が2回でも正当なケースの判断 |
| NGワード辞書に基づく我流単語 | 仕様書にないが一般的に通じる略語の判断 |
| 手順列が完全に空欄のケース | 手順の詳しさが十分かの判断 |
| - | テストケースの粒度が過去参考例と合っているか |
| - | 仕様書の網羅性 |
| - | 観点シートの構成・目次の分かりやすさ |

## 4. 基本的な使い方

### 最もシンプルな実行（NGワードなし）

```bash
python evaluate_testcases.py テスト項目書.xlsx
```

曖昧表現・複数期待値・手順省略の3項目をチェックします。
（我流単語チェックはスキップされます）

### NGワード付きの実行（推奨）

```bash
python evaluate_testcases.py テスト項目書.xlsx --keywords ng_words.txt
```

4項目すべてをチェックします。

### レポートをファイルに保存

```bash
python evaluate_testcases.py テスト項目書.xlsx --keywords ng_words.txt --output report.txt
```

標準出力に加えて、`report.txt` にもレポートを保存します。

## 5. NGワードファイルの作り方

### 形式
- テキストファイル（UTF-8）
- **TSV形式**（タブ区切り）: `NGワード<TAB>正式名称`
- `#` で始まる行はコメント
- 空行は無視

### サンプル（TH-898 ログイン制限プロジェクト）

```
# NGワードリスト（TSV形式: NG単語<TAB>正式名称）
# プロジェクトの仕様書・PRDに合わせて編集してください
ID/PW	メールアドレス・パスワード
PW	パスワード
リカバリー	パスワード再設定
```

### 作成手順

1. **仕様書・PRDを開く**
2. **正式な用語を確認する**（機能名、画面名、ボタン名、エラーメッセージ等）
3. **テストケースで使われがちな略語・造語を洗い出す**
4. NGワードファイルに `略語<TAB>正式名称` のペアを書く

### 注意点

- NGワードは**部分一致**で検索されます
  - 例: `PW` は「パスPW」にもマッチします
  - 誤検出が多い場合は、スクリプト内の `check_custom_words` を修正してください
  - TH-898では `re.search(r'(?<!パス)PW', val)` のように否定後読みで対処しました
- 短すぎるNGワード（1-2文字）は誤検出が多いため避けてください

## 6. オプション一覧

| オプション | 短縮形 | デフォルト | 説明 |
|---|---|---|---|
| `excel_file` | （位置引数） | 必須 | 評価対象のExcelファイルパス |
| `--keywords` | `-k` | なし | NGワードリスト（TSVファイル） |
| `--sheet` | `-s` | 自動検出 | テストケースシート名 |
| `--kanten` | | 自動検出 | 観点シート名 |
| `--header-row` | | 2 | ヘッダー行の行番号 |
| `--data-start-row` | | 3 | データ開始行の行番号 |
| `--output` | `-o` | なし | レポート出力先ファイルパス |

### シート名の自動検出

テストケースシートは以下の順で検索されます:
1. `テストケース_改善後` → `テストケース` → `テスト項目` → `TC`（完全一致）
2. 上記を含むシート名（部分一致）
3. 「テスト」「ケース」を含む最初のシート

観点シートは以下の順:
1. `観点` → `観点シート` → `テスト観点`

### カラムの自動検出

ヘッダー行（デフォルト: 2行目）から以下のキーワードで列を特定します:

| 内部名 | 検出キーワード |
|---|---|
| no | `No`, `NO`, `no` |
| confirm | `確認すること`, `確認項目` |
| screen | `画面`（完全一致） |
| target | `確認対象` |
| detail | `詳細` |
| steps | `手順` |
| expected | `期待値`, `期待結果` |
| notes | `備考`（完全一致優先、部分一致フォールバック） |

## 7. 評価結果の見方

### スコアの意味

| スコア | 評価 |
|---|---|
| 100点 | 運用ルール完全準拠 |
| 90-99点 | 軽微な修正で対応可能 |
| 80-89点 | 複数の修正が必要 |
| 70-79点 | 用語統一や期待値の書き直しが必要 |
| 70点未満 | 全体的な見直しが必要 |

### 減点サマリーの読み方

```
  1. 期待値の曖昧表現:     -5点 (2件検出)    ← 曖昧NGワードが2箇所
  2. 期待値の複数混在:     -4点 (2件検出)    ← 「こと」が2回以上のセルが2箇所
  3. 我流の単語:           -3点 (3件検出)    ← NGワード辞書にヒットが3箇所
  4. 手順の省略:           -0点 (0件検出)    ← 手順空欄のグループなし
```

### 修正推奨事項

減点が大きいものから優先度順に表示されます。
まず減点が大きい項目から修正するのが効率的です。

## 8. チェック項目の詳細

### チェック1: 期待値の曖昧表現（上限 -20点）

**運用ルール**: 「曖昧に書かない。『正常であること』『問題ないこと』『仕様通りであること』は基本NG」

検出するNGパターンと減点:

| パターン | 減点 |
|---|---|
| `正常であること` / `正常に動作すること` | -5点 |
| `問題ないこと` / `問題なく〜こと` | -5点 |
| `仕様通りであること` / `仕様通りに動作すること` | -5点 |
| `正しく動作すること` | -3点 |
| `適切に表示されること` / `適切であること` | -3点 |
| `正しく表示されること` | -3点 |
| `期待通り` / `想定通り` | -3点 |

**修正方法**: 具体的な画面要素・文言・状態を記載する
- NG: 「正常に動作すること」
- OK: 「ログイン後画面に遷移すること」「エラーメッセージが表示されないこと」

### チェック2: 期待値の複数混在（上限 -15点 / 1件 -2点）

**運用ルール**: 「基本的に1ケースにつき、期待値は一つ」

検出ロジック: 期待値セル内に「こと」が2回以上含まれる場合

よくあるパターンと修正例:

| パターン | 修正方法 |
|---|---|
| `〇〇すること（△△すること）` | 括弧内を本文に統合して1文にする |
| `〇〇すること、△△すること` | 別ケースに分離 or 1文に統合 |
| `〇〇すること\n△△すること` | 改行を削除して1文に統合 |

**修正例**:
- 修正前: 「表示されること（各要素が表示されていること）」
- 修正後: 「ログイン画面（メール欄・パスワード欄・ログインボタン）が表示されること」

### チェック3: 我流の単語（上限 -15点 / 1件 -1点）

**運用ルール**: 「我流の単語は使わない。要件ページやPRDに沿った表現に統一する」

NGワードファイルで定義した略語・造語を、テストケースシートと観点シートの両方から検出します。

**チェック対象**: 確認すること、画面、確認対象、詳細、手順、期待値、備考 + 観点シート全セル

### チェック4: 手順の省略（上限 -10点 / 1件 -2点）

**運用ルール**: 「手順は極力省かない」

検出ロジック:
- 同じ「確認すること」カテゴリ内で、1件でも手順が書かれていれば、そのグループの空欄は許容（グループ共有手順）
- カテゴリ内の**全ケースが空欄**の場合のみ検出

**補足**: テスト項目書では、同じカテゴリの先頭ケースに手順を書き、後続ケースは手順を省略する構造がよく使われます。この構造は正当なので、スクリプトはグループ単位で判定します。

## 10. カスタマイズ方法

### 曖昧表現パターンの追加

`evaluate_testcases.py` 冒頭の `VAGUE_PATTERNS` リストに追加:

```python
VAGUE_PATTERNS = [
    # (正規表現パターン, 減点, メッセージ)
    (r'正常であること', 5, '「正常であること」は具体的な期待値に書き換えてください'),
    # ↓ 新規追加
    (r'うまくいくこと', 5, '「うまくいくこと」は具体的な期待値に書き換えてください'),
]
```

### 減点の調整

同じく `evaluate_testcases.py` 冒頭の定数を変更:

```python
MULTI_KOTO_PENALTY = 2    # 複数こと: 1件あたりの減点
CUSTOM_WORD_PENALTY = 1   # 我流単語: 1件あたりの減点
MISSING_STEPS_PENALTY = 2 # 手順省略: 1件あたりの減点

MAX_VAGUE_PENALTY = 20         # 曖昧表現の減点上限
MAX_MULTI_KOTO_PENALTY = 15    # 複数ことの減点上限
MAX_CUSTOM_WORD_PENALTY = 15   # 我流単語の減点上限
MAX_MISSING_STEPS_PENALTY = 10 # 手順省略の減点上限
```

### ヘッダー行・データ開始行が異なるExcel

```bash
# ヘッダーが1行目、データが2行目から始まる場合
python evaluate_testcases.py テスト項目書.xlsx --header-row 1 --data-start-row 2
```

### テストケースシート名が自動検出されない場合

```bash
python evaluate_testcases.py テスト項目書.xlsx --sheet "①ログインテスト" --kanten "観点一覧"
```

## 11. トラブルシューティング

### Excelファイルが開けない
```
PermissionError: [Errno 13] Permission denied
```
→ Excelでファイルを開いている場合は閉じてから実行してください。

### テストケースシートが見つからない
```
ERROR: テストケースシートが見つかりません
```
→ `--sheet` オプションで明示的にシート名を指定してください。
   シート名一覧は実行時のログに表示されます。

### 期待値カラムが検出されない
```
WARNING: 期待値カラムが検出できません
```
→ ヘッダー行が2行目でない場合は `--header-row` で指定してください。
   ヘッダーに「期待値」「期待結果」のいずれかが含まれている必要があります。

### NGワードの誤検出が多い
→ 短いNGワード（例: `PW`）は部分一致で意図しない箇所にもヒットします。
   `evaluate_testcases.py` の `check_custom_words` 関数内で正規表現による除外を追加してください。

   例: `PW` が「パスワード(=パスPW)」にマッチしないようにする
   ```python
   # 「パス」の直後のPWは除外
   if re.search(r'(?<!パス)PW', val):
       # NGとして検出
   ```

## 付録: システムアーキテクチャ

### 全体像

```
┌───────────────────────────────────────────────────────────┐
│                    学習フェーズ                              │
│  参考例/ ──→ learn_benchmark.py ──→ benchmark.json（基準値） │
│  ※ ファイルを追加して再実行するだけで基準値が更新される       │
└───────────────────────────────────────────────────────────┘
                        │
                benchmark.json
                        ▼
┌───────────────────────────────────────────────────────────┐
│                    評価フェーズ                              │
│  評価対象.xlsx ──→ evaluate_testcases.py ──→ 100点満点      │
│    ├── チェック1〜4: 固定ルール（参考例不要）                 │
│    └── チェック5: 粒度・品質（← benchmark.json）             │
└───────────────────────────────────────────────────────────┘
```

- **チェック1〜4** は固定ルール。参考例の有無に関係なく動く
- **チェック5** だけが学習データに依存。参考例が多いほど精度向上
- benchmark.json が無くてもデフォルト値で動作（学習は任意）

### チェック5 の採点ロジック（15点満点）

| 観点 | 配点 | 判定基準 |
|------|------|----------|
| 期待値の長さ | 5点 | ideal_min未満/ideal_max超が20%超→-5, 10%超→-3, 5%超→-1 |
| 手順の詳細度 | 5点 | 記載率60%未満→-5, 80%未満→-3。番号付きステップ率20%未満→-2 |
| 末尾表現 | 5点 | 良好末尾率が基準-10%未満→-5, 基準未満→-3, 基準+10%未満→-1 |

### benchmark.json の主要構造

```json
{
  "_meta": { "source_files": [...], "total_cases": 6126 },
```
