## 1. 全体像

このシステムは「学習」と「評価」の2つのフェーズで構成されています。

```
┌─────────────────────────────────────────────────────────────┐
│                      学習フェーズ                             │
│                                                               │
│  参考例フォルダ/                                              │
│  ├── 良い例A.xlsx   ─┐                                      │
│  ├── 良い例B.xlsx    ├─→ learn_benchmark.py ─→ benchmark.json │
│  └── 良い例C.xlsx   ─┘   （分析・統計算出）    （基準値）      │
│                                                               │
│  ※ ファイルを追加して再実行するだけで基準値が更新される        │
└─────────────────────────────────────────────────────────────┘
                            │
                    benchmark.json（基準値）
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│                      評価フェーズ                             │
│                                                               │
│  評価対象.xlsx ──→ evaluate_testcases.py ──→ 評価レポート     │
│                     │                         100点満点        │
│                     ├── チェック1〜4: 運用ルール（固定ルール） │
│                     └── チェック5: 粒度・品質（←benchmark.json）│
│                                                               │
│  ※ benchmark.json があれば自動で読み込んで粒度判定に使う      │
└─────────────────────────────────────────────────────────────┘
```

### ポイント

- **チェック1〜4**（曖昧表現・複数こと・我流単語・手順省略）は**固定ルール**。参考例の有無に関係なく動く
- **チェック5**（粒度・品質）だけが**学習データに依存**する。参考例が多いほど精度が上がる
- benchmark.json が無くてもデフォルト値で動作する（学習は任意）

## 2. 学習フェーズ：参考例からベンチマークを作る

### 2-1. 学習の流れ

```
参考例フォルダに .xlsx を配置
        │
        ▼
learn_benchmark.py が各ファイルを分析
        │
        ├── シート自動検出（ヘッダー行に「期待値」「手順」を含むシートを特定）
        ├── カラム自動検出（No, 確認すること, 手順, 期待値, 備考 等）
        └── 全ケースを読み取り、以下を集計:
            │
            ├── 期待値の文字数分布（平均・5%tile・95%tile）
            ├── 手順の文字数分布
            ├── 手順ありケースの割合
            ├── 番号付きステップ（1. ① 等）の使用率
            ├── 「こと」が1回のケースの割合
            ├── 曖昧NG表現の含有率
            └── 末尾表現パターンの出現頻度（TOP15）
        │
        ▼
benchmark.json に保存
```

### 2-2. 実行方法

```bash
python learn_benchmark.py <参考例フォルダ> --output benchmark.json
```

### 2-3. 参考例として適切なファイル

| 適切 | 不適切 |
|---|---|
| レビュー済み・承認済みのテスト項目書 | 作成途中のドラフト |
| テスト実行完了済みのテスト項目書 | テンプレート（データなし） |
| QAリーダーが「良い例」と認めたもの | 不合格になったテスト項目書 |

### 2-4. benchmark.json の中身

```json
{
  "_meta": {
    "generated_at": "2026-02-26 00:07",
    "source_files": ["取引先マスタ.xlsx", "事前申請.xlsx"],

  "expected_length": {

    "p95": 39,            ← 95パーセンタイル（これより長いのは全体の5%）
    "ideal_min": 9,       ← 短すぎ判定の閾値（= p5）
```

### 2-5. learn_benchmark.py が行う分析の詳細

#### シートの自動検出

```
各シートの先頭10行をスキャンし、
「期待値」「手順」というキーワードを含む行を探す
    → 見つかった行 = ヘッダー行
    → その次の行から = データ行
    → 見つからないシートはスキップ（運用ルール、進捗 等）
```

#### カラムの自動検出

```
ヘッダー行の各セルをスキャンし、キーワードで列を特定:
  「No」「NO」          → No列
  「確認すること」       → 大分類列
  「手順」「実行手順」   → 手順列
  「期待値」「期待結果」 → 期待値列
  「備考」              → 備考列
```

#### 集計項目

| 集計項目 | 算出方法 |
|---|---|
| 期待値の文字数分布 | 全ケースの期待値セルの `len()` を収集し、平均・5%tile・95%tileを算出 |
| 手順ありケース割合 | 手順セルが空欄でも `-` でもないケースの割合 |
| 番号付きステップ率 | `1.` `①` `(1)` 等で始まる手順のケース割合 |
| こと1回率 | `re.findall(r'こと', 期待値)` が1回のケースの割合 |
| 曖昧表現率 | 「正常であること」等12パターンに一致するケースの割合 |
| 良好末尾率 | 期待値が「〜されること」「〜であること」等の定型表現で終わるケースの割合 |
| 末尾表現TOP | 末尾表現の出現頻度を集計し上位15パターンを保存 |

## 3. 評価フェーズ：ベンチマークを使って採点する

### 3-1. 5つのチェック項目

```
evaluate_testcases.py
    │
    ├── チェック1: 期待値の曖昧表現     ← 固定ルール（NGワードパターン一致）
    ├── チェック2: 期待値の複数混在     ← 固定ルール（「こと」の出現回数）
    ├── チェック3: 我流の単語           ← NGワード辞書（ユーザーが用意）
    ├── チェック4: 手順の省略           ← 固定ルール（グループ単位の空欄検出）
    └── チェック5: 粒度・品質           ← ★ benchmark.json の基準値を使用
```

### 3-2. チェック5 の採点ロジック（15点満点）

benchmark.json の基準値と評価対象を比較して採点します。

```
(1) 期待値の長さ（5点）
    評価対象の期待値を全件チェック
    benchmark.json の ideal_min 未満 → 「短すぎ」カウント
    benchmark.json の ideal_max 超 → 「長すぎ」カウント
    短すぎ+長すぎ が全体の:
      20%超 → -5点
      10%超 → -3点
       5%超 → -1点

(2) 手順の詳細度（5点）
    手順記載率 = グループ構造を考慮した手順ありケースの割合
    benchmark.json の steps_coverage.warning (60%) 未満 → -5点
    benchmark.json の steps_coverage.good (80%) 未満    → -3点
    上記を満たしても、番号付きステップ率が
    benchmark.json の numbered_steps.good (20%) 未満    → -2点

(3) 末尾表現（5点）
    benchmark.json の good_endings.patterns と末尾一致するケースの割合
    benchmark.json の good_endings.good (80%) -10%未満 → -5点
    benchmark.json の good_endings.good (80%) 未満     → -3点
    benchmark.json の good_endings.good +10% 未満      → -1点
```

### 3-3. benchmark.json の読み込み優先順位

```
1. --benchmark オプションで指定されたパス
2. evaluate_testcases.py と同じフォルダの benchmark.json
3. どちらも無い場合 → デフォルト値を使用（学習なしでも動作する）
```

## 4. 学習データが評価に反映される仕組み（詳細）

### 具体例：期待値の長さ判定

```
学習データ無しの場合:
  ideal_min = 10（ハードコードのデフォルト）
  ideal_max = 80（ハードコードのデフォルト）

learn_benchmark.py 実行後:
  参考例6,072件の期待値を分析
  5パーセンタイル = 9字
  95パーセンタイル = 39字
  → ideal_min = max(5, 9) = 9
  → ideal_max = max(80, 39) = 80
  → 参考例で9字未満は全体の5%しかない = 9字以上が「普通」
```

### 具体例：良好な末尾表現パターン

```
学習データ無しの場合:
  固定12パターン（手動で設定）:
    されていること, されること, していること, できること, ...

learn_benchmark.py 実行後:
  参考例6,072件の末尾表現を集計し、出現頻度TOP15を抽出:
    1位: されていること (1,002回)
    2位: されること (907回)
    3位: していること (823回)
    ...
  → 参考例で実際に使われているパターンが基準になる
  → 新しい参考例を追加すると、そのプロジェクト特有の表現も基準に入る
```

### 具体例：手順記載率の基準

```
学習データ無しの場合:
  good = 80%（ハードコード）

learn_benchmark.py 実行後:
  参考例A（大規模）: 77.4%
  参考例B（バグ修正）: 100%
  加重平均 = 77.0%
  → good = 80%（JSONの値 or デフォルト）

  もし今後「手順記載率50%」の参考例が追加されても、
  good/warning の閾値は benchmark.json で直接編集可能
```

## 5. 参考例を増やすと何が変わるか

### 現在の学習データ（参考例2件）

| 指標 | 値 | 信頼性 |
|---|---|---|
| 期待値 平均文字数 | 21.1字 | 低い（大規模1件に偏り） |
| 手順記載率 | 77.0% | 低い（大規模1件に偏り） |
| こと1回率 | 98.9% | 高い（6,000件のデータ） |
| 良好末尾パターン | 15パターン | 中程度 |

### 参考例を5件に増やした場合（想定）

| 指標 | 値 | 信頼性の変化 |
|---|---|---|
| 期待値 平均文字数 | 18〜25字（安定） | 高い（規模の異なる5件で平均化） |
| 手順記載率 | 70〜85%（安定） | 高い（バグ修正/新機能/リグレッション混合） |
| こと1回率 | 98〜99%（安定） | 非常に高い |
| 良好末尾パターン | 20+パターン | 高い（ドメイン特有の表現も含む） |

### 増やすと特に効果が大きいもの

| 増やす参考例の種類 | 効果 |
|---|---|
| **同規模・同種のテスト項目書** | 期待値の文字数・手順の詳しさの基準が正確になる |
| **異なるドメインのテスト項目書** | 末尾表現パターンが増え、誤検出が減る |
| **レビューでNGになった→修正したもの** | 修正前の特徴がNG基準として使える（将来拡張） |

## 6. 運用手順

### 初回セットアップ

```
Step 1: 参考例フォルダを作る
  mkdir C:\Users\池田尚人\ClaudeCode用\テストケース評価自動化\参考例

Step 2: レビュー済みの良いテスト項目書をコピーする
  参考例/
  ├── テスト項目書_取引先マスタ.xlsx
  └── テスト項目書_事前申請.xlsx

Step 3: ベンチマークを生成
  python learn_benchmark.py 参考例/ --output benchmark.json

Step 4: benchmark.json を evaluate_testcases.py と同じフォルダに置く
  （すでに同フォルダなら不要。自動読み込みされる）
```

### 参考例を追加する時

```
Step 1: 参考例フォルダにファイルを追加
  参考例/
  ├── テスト項目書_取引先マスタ.xlsx
  ├── テスト項目書_事前申請.xlsx
  └── テスト項目書_新案件.xlsx  ← 追加

Step 2: learn_benchmark.py を再実行
  python learn_benchmark.py 参考例/ --output benchmark.json

  → benchmark.json が更新される
  → 次回の評価から自動的に新しい基準値が使われる
```

### 日常の評価

```
python evaluate_testcases.py 対象.xlsx --keywords ng_words.txt

  → benchmark.json があれば自動読み込み
  → なくてもデフォルト値で動作
```

### Claudeに依頼する場合

```
このテスト項目書を評価して
C:\path\to\テスト項目書.xlsx
```

Claudeは CLAUDE.md の指示に従い、evaluate_testcases.py を実行する。
benchmark.json が同フォルダにあれば自動で使われる。

## 7. ファイル構成

```
evaluate_testcases.py と同じフォルダ（最新版/）
│
├── evaluate_testcases.py    評価スクリプト本体
│   ├── チェック1〜4: 固定ルール（スクリプト内にハードコード）
│   └── チェック5: benchmark.json の値を使って粒度判定
│
├── learn_benchmark.py       学習スクリプト
│   └── 参考例フォルダの .xlsx を全件分析し benchmark.json を生成
│
├── benchmark.json           学習結果（基準値）
│   └── evaluate_testcases.py が同フォルダから自動読み込み
│
├── ng_words_sample.txt      NGワードサンプル（プロジェクトごとに作成）
│
├── テストケース評価ガイド.md          人間がスクリプトを直接使う時の手引き
├── AI向け評価ツール利用ガイド.md      Claudeに評価を依頼する時の手引き
└── 学習と評価の仕組み.md              この文書

参考例フォルダ（任意の場所）
├── 良い例A.xlsx
├── 良い例B.xlsx
└── 良い例C.xlsx  ← 追加するだけでOK

~/.claude/
├── CLAUDE.md                全チャット共通のAI指示
└── projects/.../memory/
    ├── MEMORY.md            メモリ索引
    ├── testcase_evaluation.md   評価ロジック詳細
    └── test_item_format.md      テスト項目書の書き方ルール
```

### データの流れ（まとめ）

```
参考例.xlsx ──→ learn_benchmark.py ──→ benchmark.json
                 （学習）                   │
                                            │ 自動読み込み
                                            ▼
評価対象.xlsx ──→ evaluate_testcases.py ──→ 評価レポート（100点満点）
                  + ng_words.txt
                  + benchmark.json
```
