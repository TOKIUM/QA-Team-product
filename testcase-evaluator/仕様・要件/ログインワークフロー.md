# ログインワークフロー（invoicing-staging.keihi.com/login 起点）

**作成日**: 2026-03-03
**対象環境**: dev / invoicing-staging
**根拠**: 全て実機確認済み（Playwright自動調査）

---

## 全体フロー概要

```
invoicing-staging.keihi.com/login
  │
  ├─ A. invoicing直接ログイン
  │
  ├─ B. TOKIUM IDでログイン
  │     │
  │     ▼
  │   dev.keihi.com/users/sign_in
  │     ├─ B-1. TOKIUM ID メール+PWログイン
  │     ├─ B-2. Googleでログイン
  │     ├─ B-3. Microsoft 365でログイン
  │     └─ B-4. サブドメイン経由ログイン
  │           │
  │           ▼
  │         dev.keihi.com/subdomains/input
  │           │
  │           ▼
  │         [sd].dev.keihi.com/subdomains/sign_in
  │
  ├─ C. パスワードを忘れた場合
  │
  └─ D. 新規登録
```

---

## A. invoicing直接ログイン

### A-正常系: ログイン成功

```
[invoicing/login]
  │ メールアドレス + パスワード入力
  │ [ログイン] ボタン押下
  ▼
[invoicing/invoices]  ← 請求書一覧
```

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | invoicing/login にアクセス | ログイン画面表示 |
| 2 | メールアドレスを入力 | - |
| 3 | パスワードを入力 | - |
| 4 | [ログイン] ボタン押下 | invoicing/invoices に遷移 |

### A-異常系

| # | 条件 | エラーメッセージ | 表示方式 | スクリーンショット |
|---|------|----------------|---------|-----------------|
| A-E1 | メールアドレス空欄で送信 | 「必須項目です」 | 入力欄直下（赤テキスト）+ 入力欄が赤枠 | err_01_invoicing_empty.png |
| A-E2 | 未登録メールアドレス + 任意のPW | 「メールアドレスまたはパスワードが間違っています。再度お試しください。」 | PW欄直下（赤テキスト）+ 右上トースト通知 | err_02_invoicing_invalid.png |
| A-E3 | 登録済みメール + 不正パスワード | 「メールアドレスまたはパスワードが間違っています。再度お試しください。」 | 同上 | err_03_invoicing_wrong_pw.png |

> **特徴**: invoicing側はフロントエンドバリデーション（React）。空欄はクライアント側で即時検知、認証失敗はサーバー応答後に表示。

## B. TOKIUM IDでログイン

### 共通フロー: invoicing → TOKIUM IDログイン画面

```
[invoicing/login]
  │ [TOKIUM IDでログイン] ボタン押下
  ▼
[invoicing/auth-redirect]
  │ 自動リダイレクト
  ▼
[dev.keihi.com/users/sign_in]  ← TOKIUM IDログイン画面
```

---

### B-1. TOKIUM ID メール+PWログイン

#### B-1-正常系: ログイン成功（SSO経由でinvoicingに戻る）

```
[dev.keihi.com/users/sign_in]
  │ メール + PW入力 → [ログイン]
  ▼
[invoicing/auth/callback?token=...]  ← SSOコールバック
  │ トークン処理・セッション作成
  ▼
[invoicing/invoices]  ← 請求書一覧
```

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | invoicing/login で [TOKIUM IDでログイン] 押下 | dev.keihi.com/users/sign_in に遷移 |
| 2 | メールアドレスを入力 | - |
| 3 | パスワードを入力 | - |
| 4 | [ログイン] ボタン押下 | SSOコールバック経由でinvoicing/invoices に遷移 |

> **注意**: invoicing経由のTOKIUM IDログインは、dev.keihi.com/transactionsを経由せず、SSOコールバックでinvoicingに直接戻る。

#### B-1-異常系

| # | 条件 | エラーメッセージ | 表示方式 | スクリーンショット |
|---|------|----------------|---------|-----------------|
| B-1-E1 | 空欄で送信 | 「ログインに失敗しました」 | ページ本文（プレーンテキスト、CSSクラスなし） | err_05_tokiumid_empty.png |
| B-1-E2 | 不正パスワード | 「ログインに失敗しました」 | 同上 | err_04_tokiumid_wrong_pw.png |
| B-1-E3 | サブドメイン必須アカウントで直接ログイン | 「許可されていないログイン方法です。サブドメインを入力してからログインしてください。」 | ページ本文（赤テキスト） | err_08_subdomain_required.png |

> **重要**: B-1-E3 は `sso_only_login_enabled_from` が設定されたテナントのユーザーに発生。正しいメール+PWでもサブドメイン経由でないと拒否される。

### B-2. Googleでログイン

#### B-2-正常系

```
[dev.keihi.com/users/sign_in]
  │ [Googleでログイン] ボタン押下
  ▼
[POST dev.keihi.com/omniauth/google_oauth2]
  │ Google OAuth2認証画面にリダイレクト
  ▼
[accounts.google.com/...]  ← Googleアカウント選択/認証
  │ 認証成功
  ▼
[dev.keihi.com/transactions]  ← 経費一覧
```

> **注意**: invoicing/login 経由で到達した場合のリダイレクト先はSSOコールバック経由でinvoicingに戻る可能性あり（未実機確認）。

---

### B-3. Microsoft 365でログイン

#### B-3-正常系

```
[dev.keihi.com/users/sign_in]
  │ [Microsoft 365でログイン] ボタン押下
  ▼
[POST dev.keihi.com/omniauth/microsoft_office365]
  │ Microsoft認証画面にリダイレクト
  ▼
[login.microsoftonline.com/...]  ← Microsoftアカウント認証
  │ 認証成功
  ▼
[dev.keihi.com/transactions]  ← 経費一覧
```

### B-4. サブドメイン経由ログイン

#### B-4-正常系: サブドメイン入力 → ログイン成功

```
[dev.keihi.com/users/sign_in]
  │ [サブドメインを入力] ボタン押下
  ▼
[dev.keihi.com/subdomains/input]  ← サブドメイン入力画面
  │ サブドメイン入力（例: th-01）→ [送信]
  ▼
[th-01.dev.keihi.com/subdomains/sign_in]  ← サブドメインログイン画面
  │ メール + PW入力 → [ログイン]
  ▼
[th-01.dev.keihi.com/transactions]  ← テナント経費一覧
  フラッシュ:「ログインしました。」
```

| ステップ | 操作 | 期待結果 |
|---------|------|---------|
| 1 | TOKIUM IDログイン画面で [サブドメインを入力] 押下 | dev.keihi.com/subdomains/input に遷移 |
| 2 | サブドメイン入力（例: th-01） | 送信ボタンが有効化される |
| 3 | [送信] ボタン押下 | th-01.dev.keihi.com/subdomains/sign_in に遷移 |
| 4 | メールアドレスを入力 | - |
| 5 | パスワードを入力 | - |
| 6 | [ログイン] ボタン押下 | th-01.dev.keihi.com/transactions に遷移。「ログインしました。」表示 |

#### B-4-異常系

**サブドメイン入力画面（dev.keihi.com/subdomains/input）**

| # | 条件 | エラーメッセージ | 表示方式 | スクリーンショット |
|---|------|----------------|---------|-----------------|
| B-4-E1 | 空欄のまま | （送信不可） | 送信ボタンがdisabled（グレーアウト） | 05_subdomain_empty_disabled.png |
| B-4-E2 | 無効なサブドメイン入力 | 「無効なサブドメインです」 | 入力欄上部（赤テキスト、⚠アイコン付き） | 04_subdomain_error.png |

**サブドメインログイン画面（[sd].dev.keihi.com/subdomains/sign_in）**

| # | 条件 | エラーメッセージ | 表示方式 | スクリーンショット |
|---|------|----------------|---------|-----------------|
| B-4-E3 | 空欄で送信 | 「ログインに失敗しました」 | ページ本文（プレーンテキスト） | err_07_subdomain_empty.png |
| B-4-E4 | 不正パスワード | 「ログインに失敗しました。あと○回失敗すると、アカウントがロックされます。パスワード再設定を行なってください。」 | ページ本文（赤テキスト、⚠アイコン付き） | err_06_subdomain_wrong_pw.png |
| B-4-E5 | sso_only制限 + ID/PWログイン | 「この事業所ではTOKIUM IDでのログインのみ許可されています。TOKIUM IDでログインしてください。」 | フラッシュメッセージ（※未実機確認、TH-898仕様） | - |

> **B-4-E4 補足**: 失敗回数カウントはサーバー側で管理。残り回数が0になるとアカウントロックされる。

## C. パスワードを忘れた場合

### invoicing側

```
[invoicing/login]
  │ [パスワードを忘れた場合] リンク押下
  ▼
[invoicing/recovery]  ← invoicingのPW回復画面
```

### TOKIUM ID側

```
[dev.keihi.com/users/sign_in]
  │ [パスワードを忘れた方はこちら] リンク押下
  ▼
[dev.keihi.com/users/password/new]  ← パスワード再発行画面
  │ メールアドレス入力 → [パスワードを再発行]
  ▼
  パスワード再発行URLをメール送信
```

| 画面 | リンクテキスト | 遷移先 |
|------|-------------|-------|
| invoicing/login | 「パスワードを忘れた場合」 | invoicing/recovery |
| dev.keihi.com/users/sign_in | 「パスワードを忘れた方はこちら」 | dev.keihi.com/users/password/new |
| [sd].dev.keihi.com/subdomains/sign_in | 「パスワードを忘れた方はこちら」 | [sd].dev.keihi.com/users/password/new |
