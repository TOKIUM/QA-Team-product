1. 各項目の仕様とテスト要件の整理
■ サブドメイン(TOKIUM IDを使用する場合のみ設定可能)
• 仕様: SAMLを使用するためには、サブドメインが必須となります。ただし、SAML以外のログイン方式（ID/PWなど）でもサブドメインは利用される可能性があります。
• テスト要件: すべての認証パターンで網羅する必要はありませんが、テスト全体のどこかで**「最低1回はサブドメインあり」**の検証を含める必要があります。
■ IP制限
• 仕様: 通常のログインにおいては、IP制限の「あり/なし」が設定されます。ただし、SAMLを使用する場合に限り、**「IP制限を無視するオプション設定」**が存在します。
• テスト要件: 基本的なIP制限の「あり/なし」は、全体のどこかで確認できれば問題ありません。しかし、SAMLのテストにおいては、この無視オプションの挙動を確認するためのパターンを追加する必要があります。
■ SAML
• 仕様: 開発側への確認の結果、**「SAMLとSSOはシステム上同じ扱い（振る舞いが変わらない）」**であることが判明しています。（※会話履歴より）
• テスト要件: SAMLとSSOが同じ扱いであるため、既存の「SSO限定」でエラーになるパターンのうち1つを、SAMLのテストケースに置き換えます。その上で、SAML特有のIP制限無視オプションの検証パターン（プラス3パターン程度）を新たに追加します。

2. テストケース（デシジョンテーブル）への具体的な反映手順
上記を踏まえ、既存のデシジョンテーブルを以下の手順で修正します。
• 手順1：SAMLパターンの作成（置き換え） 既存の「SSO限定」で弾かれる「列2」または「列6」のどちらか1つにおいて、条件「SAML」を「あり」に変更し、SAMLの検証用ケースとします。
• 手順2：サブドメイン要件のクリア 手順1でSAML用にした列の条件「サブドメイン」を必ず「あり」にします（SAMLの必須条件のため）。これにより「最低1回はサブドメインありを含める」という要件も同時に達成できます。
• 手順3：SAML特有のIP制限オプションパターンの追加 条件行に「IP制限無視オプション」を新設します。その上で、SAML用の列を3つほど追加し、以下の組み合わせでオプションの挙動を網羅します。
    ◦ SAML: あり / サブドメイン: あり / IP制限: なし / 無視オプション: OFF
    ◦ SAML: あり / サブドメイン: あり / IP制限: あり / 無視オプション: OFF
    ◦ SAML: あり / サブドメイン: あり / IP制限: あり / 無視オプション: ON
• 手順4：通常のIP制限パターンの割り当て デシジョンテーブルの空いている列（通常のID/PWログインのケースなど）を利用して、条件「IP制限」の行に「あり」「なし」を割り当て、通常のIP制限が機能するかを確認します。
このように整理・修正することで、会議で出たすべての要望と仕様を満たしたテストケースを作成することが可能です。
