## 1. 機能テスト観点
以下の各テストについて、今回のテスト対象（修正点）が関係あるかどうかを熟考する。
関係ない場合は、理由とともに「対象外」の旨を残す。
関係ある場合は、具体的な観点を検討する。

### 1-1. 仕様のチェック
必須項目
- [ ] 仕様に記載の挙動が仕様通りに実装されていることの確認
- [ ] 要件に記載されていることが実現されていることの確認
- [ ] 正常系：仕様で想定されている入力に対して仕様の記載通りの結果になること
- [ ] 異常系：想定されていない入力に対してエラー理由が分かる表示とともにエラーになるか、そもそも実行できないこと

### 1-2. 仕様の妥当性チェック
必須項目
- [ ] 仕様通りに実装されていたとしても実際のユーザー目線では不親切な場合がある。そのような仕様不備がないか確認する

### 1-3. 入力値テスト
<!-- チームの経験から追記してください -->
- [ ] 正常値（ひらがな・全角カタカナ・半角カタカナ・数値・数値(マイナス)・数値(小数)・漢字・全角アルファベット・半角アルファベット）
- [ ] 境界値（最小・最大・最小-1・最大+1）
- [ ] 異常値（型違い・空・null・特殊文字・SQL injection・XSS）
- [ ] 桁数制限（上限ぴったり・上限+1）
- [ ] 禁則文字・フォーマット制約（仕様に定義されている場合はその境界値チェック）
- [ ] 金額入力欄が存在する部分が対象の場合、金額計算（特に小数点の切捨て・切上げ、按分計算）は **必ずP1で** テストケース化すること

### 1-4. 認証・認可
<!-- このプロダクト固有の注意点を追記してください -->
- [ ] ログイン(メールアドレス認証・SSO・SAML認証・代理ログイン・テナント切り替え)・ログアウト正常系
- [ ] セッションタイムアウト
- [ ] 権限のないページへの直接URLアクセス
- [ ] ロール別の表示・操作権限の違い
- [ ] 複数タブ・複数デバイスでの同時ログイン
- [ ] ログインセッションの維持(SaaSにログインしているセッションがある場合、AIエージェントへのログイン操作を開始したらリダイレクトされてログイン操作がスキップされる)
- [ ] 不正アクセスの検知（ログに記録されること）
- [ ] 権限の定義通りの範囲で操作できること（許可された操作のみ実行可能、それ以外は拒否）

### 1-5. データ整合性
- [ ] 登録→参照→更新→削除のCRUD一連フロー
- [ ] 関連データへの波及確認
- [ ] 同時更新（排他制御）
- [ ] トランザクション途中でのエラー・ロールバック

### 1-6. 画面遷移・UI
- [ ] 正常な画面遷移フロー
- [ ] ブラウザバック・リロードの挙動
- [ ] 二重送信防止
- [ ] バリデーションエラー時のメッセージ表示
- [ ] 必須項目未入力

### 1-7. 高頻度
- [ ] ボタン押下を短時間に連続実行する(そもそも連続実行できないこと・もしできる場合は重複実行されないこと)
- [ ] 操作の競合（複数ユーザーが同一リソースを同時操作）

### 1-8. 英語化対応
- [ ] 新規作成画面の英語化対応(設定からEnglishに切り替えたら画面全体が正しく英語翻訳されている)
- [ ] 変更箇所の英語化対応(設定からEnglishに切り替えたら画面全体が正しく英語翻訳されている)
- [ ] 新規ボタン、表示文言、エラーメッセージが英語化されていること

### 1-9. 外部連携API
- [ ] 仕様書通りにAPIを使用して情報の取得ができること
- [ ] 仕様書通りにAPIを使用して情報の登録ができること
- [ ] 既存の外部連携APIの機能が維持されていること
- [ ] API仕様に変更がある場合、変更後に同等の機能が維持されていること

### 1-10. 契約変更
- [ ] オプション契約の状況に応じた表示内容の変化
- [ ] 操作不可能な機能に対するバリデーション
- [ ] 開発機能がすべての契約形態の事業所への影響がないか

### 1-11 既存機能への影響確認
- [ ] 新機能の追加・修正によって既存機能の挙動が維持されていること(影響範囲を調べて実施)
- [ ] 影響範囲の詳細は「5. 既存機能への影響確認 影響範囲チェックリスト」を参照

### 1-12. シナリオテスト
- [ ] 想定されるユースケースを通るシナリオを設定し、そのシナリオを通す中で不便な点、エラーがないことを確認

### 1-13. 金額確認
- [ ] 手入力の金額が正しく計算されること
- [ ] 自動入力（OCR等）の金額が正しく計算されること
- [ ] 自動仕訳の金額が正しいこと
- [ ] 定期支払値が正しいこと
- [ ] 整数、負の値、小数それぞれの金額で正しい結果になること
- [ ] 丸め誤差が発生するケース・発生しないケースの両方を確認
- [ ] 割合で按分する金額計算の正確性
- [ ] 消費税額の自動計算が正しいこと
- [ ] 源泉徴収税額の自動計算が正しいこと
- [ ] 税抜・税込設定の切替で金額が正しく変動すること
- [ ] 金額計算に関係する修正が入った場合・裏側のコードが大幅に変わった場合は **必ずP1で** 網羅確認すること

### 1-14. 時間経過系
- [ ] 日跨ぎ・月跨ぎ・年跨ぎ・年度跨ぎで挙動が変わる機能の確認
- [ ] 指定時刻経過後のステータス変化（有効期限切れ、バッチ処理のトリガー等）
- [ ] 長期間経過しても機能が維持されること

### 1-15. リリースによるデータ移行の影響確認
- [ ] 意図せぬ設定変更が発生していないこと（設定値の削除、不要な設定の増加）
- [ ] リリース前に作成していた設定・データをリリース後に操作できること
- [ ] リリース前に利用できていた会計フォーマットが継続利用可能であること
- [ ] ロールバックが発生した時、元の状態に復帰できること
- [ ] リリース前に申請開始しておき、承認前にリリースされたときに影響がないか

### 1-16. 互換性確認（ファイルフォーマット）
- [ ] 利用可能フォーマットのデータを処理できること
- [ ] 対象外フォーマットのデータは処理しないこと（エラー表示）
- [ ] 画像ファイルの各拡張子(jpg, png, pdf等)で正常動作すること
- [ ] インポートファイルの各拡張子(csv, xlsx等)で正常動作すること
- [ ] ファイルフォーマット変更を伴う開発の場合、既存フォーマットのファイルがそのまま使えること

## 2. 非機能テスト観点
以下の各テストについて、今回のテスト対象（修正点）が関係あるかどうかを熟考する。
関係ない場合は、理由とともに「対象外」の旨を残す。
関係ある場合は、具体的な観点を検討する。

### 2-1. パフォーマンス・負荷
<!-- 過去に問題になった事例があれば追記してください -->
- [ ] 大量データ時の応答速度(表示対象のデータが大量にある・選択肢が大量にある)
- [ ] 同時アクセス時の挙動
- [ ] タイムアウト設定値の確認
- [ ] 大容量ファイル（画像等）のアップロード処理
- [ ] 大量ファイルの一括アップロード処理
- [ ] インポートファイルのサイズ大・処理件数膨大な場合の挙動
- [ ] 参照するマスタ（部署、プロジェクト等）の件数が膨大な場合の挙動

### 2-2. 互換性
- [ ] 対応ブラウザ（Chrome最新・Edge最新・Safari最新・Firefox最新）
- [ ] 画面サイズ（PC・タブレット・スマートフォン）
- [ ] 対応OS（Windows・mac・Android・iOS）
- [ ] アプリ対応（Android・iOS・Windowsタブレット）：新機能追加時にアプリ版でもWEB版と同等の機能追加があること

### 2-3. セキュリティ
- [ ] 機密情報のURLパラメータ露出
- [ ] エラーメッセージへの内部情報露出
- [ ] ログへの個人情報出力

### 2-4. UX
- [ ] 画面全体に崩れがないか
- [ ] カラム幅や画像領域など、調整可能な領域のサイズ変更ができるか。実行後に崩れがないか
- [ ] ユースケースに則って作業する際、使いにくさ（無駄な視線移動、無駄にマウスの移動が必要、など）がないか
- [ ] 画面内の説明が十分であること（経験が少ないユーザーでも理解できる）
- [ ] 既存画面・既存機能からの変化が小さいこと（大きい場合は事前通知が必要）

### 2-5. 統一性
- [ ] 複数プロダクト間でのデザイン統一性
- [ ] 複数プロダクト間での操作感統一性
- [ ] Web-スマホアプリ間での操作感統一性
- [ ] 複数プロダクト間、クライアント機器間での文言表記の統一

### 2-6. お客様提供ドキュメント
- [ ] ヘルプ文言がわかりやすいこと
- [ ] マニュアルの内容が正確であること（画面キャプチャ、手順が最新の仕様と一致していること）

### 2-7. 本番環境固有の確認
- [ ] 本番環境固有のオブジェクトや機能が影響を受けないこと
- [ ] サポート用チャットボタンや機能追加通知バナーなど、本番のみに表示される要素が正常に動作すること

## 3. このプロダクト固有の注意ポイント
<!-- QAチームが経験から学んだ、このプロダクト特有の注意点を記載してください -->

### よく見落とすポイント
- （例）〇〇機能と△△機能が連携している場合、片方を変更すると影響が出やすい
- アプリの影響確認を見落としがち（経費の作成、申請、承認・支払依頼申請の承認・カード連携はアプリも影響範囲）
- 何らかのマスタを更新・削除した場合、そのマスタを引用して過去に作成した経費や請求書のデータで異常が出ることがある（プロジェクト設定から「プロジェクトA」を削除→過去に「プロジェクトA」を選択して作成した経費データが壊れるなど）
- 経費、請求書に新たな入力フィールドが追加された場合、各種検索画面や、集計画面、会計データ定期出力の設定で異常が発生する

### 過去バグから得た観点
- （例）日付入力は和暦・西暦の混在に注意
- 経理業務に関連するプロダクトなので、金額計算は非常に重視している
- 小数点切り捨て・切り上げ、費用負担部署を割合で指定するといった、小数点が発生しうる金額計算は、バグ発生の可能性が高く、かつ発生してしまった場合の緊急度が高い

### 外部連携・API
- （例）決済APIのレスポンス遅延時の挙動確認を忘れない
- APIでの情報取得は「画面上で検索できる情報はAPIでも使える」が基本線。従って、画面上に検索条件が増えた場合はAPI側にもパラメータ追加したい

## 5. 既存機能への影響確認 影響範囲チェックリスト

1-11「既存機能への影響確認」を実施する際に、変更の種類に応じて以下を確認する。

### 5-1. 画面内に項目の追加が発生する場合
- [ ] その画面全体で、仕様変更前の挙動が維持されていること
- [ ] その画面全体に表示崩れがないこと
- [ ] リリース前に作成済みの設定やデータが今まで通り利用でき、新たに更新もできること
- [ ] 新規項目が追加されたコンポーネントの他項目が、仕様変更前の挙動を維持していること
- [ ] APIで情報の取得、設定ができること

### 5-2. 計算ルールに変更が発生する場合
- [ ] そのオブジェクトにおいて発生しうる計算の演算パターンを網羅確認
- [ ] 会計データ出力した結果、その数値が正しいこと

### 5-3. 経費データに変更が発生した場合
- [ ] 経費の新規作成が全パターンで成功すること
    - 手動作成
    - 自動作成（webから登録、アプリから登録）
    - カード連携による登録（私用カード、法人カード（会社利用/個人利用））
- [ ] 経費の統合が正常に動作すること
- [ ] 経費のオプション設定が全て正常に利用できること（全ON/全OFF/抜粋ON）
- [ ] 添付画像が正常に登録できること（対応拡張子全種、表裏添付、削除、入れ替え、タイムスタンプ）
- [ ] 変更履歴が正常に記録されること（全項目で履歴が残ること）
- [ ] APIで情報の取得、設定ができること
- [ ] アプリで情報の操作、取得、表示ができること

### 5-4. 請求書データに変更が発生した場合
- [ ] 請求書の新規作成が全パターンで成功すること
    - 手動作成（個別、一括登録）
    - 自動作成（web一括登録、メール（識別IDあり/なし）、一般アップローダー（識別IDあり/なし）、管理者用アップローダー（識別IDあり/なし））
    - 国税関係書類から移動（手動/自動）
    - 自動入力中画面から移動（手動）
- [ ] 関連書類の設定が正常に動作すること（新規設定、既存設定の変更）
- [ ] 請求書のオプション設定が全て正常に利用できること（全ON/全OFF/抜粋ON）
- [ ] 添付画像が正常に登録できること（対応拡張子全種、10枠目まで添付、削除、入れ替え、タイムスタンプ）
- [ ] 明細一覧が正常に動作すること（追加、削除、変更、移動、明細インポート、過去明細から作成、過去明細で上書き）
- [ ] 変更履歴が正常に記録されること（全項目で履歴が残ること）
- [ ] 定期支払機能で請求書が作成できること
- [ ] APIで情報の取得、設定ができること
- [ ] アプリで情報の取得、表示ができること

### 5-5. 国税関係書類データに変更が発生した場合
- [ ] 国税関係書類の新規作成が全パターンで成功すること
    - 手動作成（個別、一括登録）
    - 自動作成（web一括登録、メール（識別IDあり/なし）、一般アップローダー（識別IDあり/なし）、管理者用アップローダー（識別IDあり/なし））
    - 自動入力中画面から移動（手動）
- [ ] 関連書類の設定が正常に動作すること（新規設定、既存設定の変更）
- [ ] 国税関係書類のオプション設定が全て正常に利用できること（全ON/全OFF/抜粋ON）
- [ ] 添付画像が正常に登録できること（対応拡張子全種、10枠目まで添付、削除、入れ替え、タイムスタンプ）
- [ ] 変更履歴が正常に記録されること（全項目で履歴が残ること）
- [ ] APIで情報の取得、設定ができること

### 5-6. 権限に変更が発生した場合
- [ ] ユーザーの新規作成ができること（権限個別設定、全権限設定、選択的抜粋）
- [ ] ユーザーの変更ができること（権限個別設定、全権限設定、選択的抜粋）
- [ ] リリース前に存在していたユーザーについて動作チェックすること
    - 既存の権限状態が維持されており、不正な操作はできないこと
    - 設定変更した場合は、設定変更後の制限に従った挙動になること
- [ ] APIで情報の取得、設定ができること
