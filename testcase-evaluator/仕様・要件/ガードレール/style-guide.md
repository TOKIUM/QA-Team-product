## テスト観点ドキュメントの構成

テスト観点は計画段階のドキュメントであり、チェックリスト形式を維持する。

```
# [機能名] テスト観点
## メタ情報
- 対象機能:
- 作成日:
- 作成者: Claude (QA自動化)
- レビュアー:
- ステータス: Draft / Reviewed / Approved

## 1. テスト対象の概要
（機能の概要を2〜3行で）

## 2. テスト観点一覧
### 2-1. 機能観点
### 2-2. 非機能観点
### 2-3. リグレッション観点

## 3. テスト対象外
（スコープ外を明記することでレビュー効率UP）

## 4. 特記事項・リスク
```

## テストケースドキュメントの構成

テストケースはGherkin形式で記述する。
自動テスト（Playwright / Vibium）への変換を前提とした構造にする。

### ドキュメント全体構造

```
# [機能名] テストケース

## メタ情報
- 対象機能:
- 作成日:
- 作成者: Claude (QA自動化)
- レビュアー:
- ステータス: Draft / Reviewed / Approved
- 参照テスト観点:

---

## Feature: [機能グループ名]

### TC-XXX-001 @P1 @[観点タグ]

**Scenario**: [シナリオの要約（1行）]

  Given [前提条件]
    And [追加の前提条件]
  When [操作]
    And [追加の操作]
  Then [期待結果]
    And [追加の期待結果]

**結果**: -
**備考**: -
```

## Gherkin記述ルール

### タグ
- 優先度: `@P1` `@P2` `@P3`
- 観点: `@権限` `@金額計算` `@画面遷移` `@データ整合性` `@仕様` `@異常系` 等
- タグは見出し行に記載する: `### TC-XXX-001 @P1 @権限`

### Given（前提条件）
文末は「〜している」「〜である」「〜済みである」で統一する。

| パターン | 書き方 |
|---------|--------|
| ログイン状態 | `Given [ロール名]でサインインしている` |
| データ存在 | `Given [対象]が[N件]存在する` |
| 設定状態 | `Given [設定名]が[値]に設定されている` |
| DB操作前提 | `Given DB操作で[対象]が[状態]である` |

### When（操作）
文末は「〜する」で統一する。操作対象のUI要素は「」で囲む。

| パターン | 書き方 |
|---------|--------|
| クリック | `When 「〇〇」をクリックする` |
| 入力 | `When 「〇〇」に「△△」を入力する` |
| 選択 | `When 「〇〇」から「△△」を選択する` |
| 画面遷移 | `When /path にアクセスする` |
| ボタン押下 | `When 「〇〇」ボタンを押下する` |
| APIリクエスト | `When [METHOD] /api/xxx にリクエストを送信する` |
| 待機 | `When [N]秒待機する` |

### Then（期待結果）
文末は「〜される」「〜である」「〜こと」で統一する。

| パターン | 書き方 |
|---------|--------|
| 表示確認 | `Then 「〇〇」が表示される` |
| 非表示確認 | `Then 「〇〇」が表示されない` |
| 遷移確認 | `Then /path に遷移する` |
| 値確認 | `Then 「〇〇」欄に「△△」が表示される` |
| 状態確認 | `Then ステータスが「〇〇」である` |
| 数値確認 | `Then [項目名]が[計算式 = 値]である` |
| APIレスポンス | `Then HTTPステータス [コード] が返却される` |
| エラー確認 | `Then 「〇〇」というエラーメッセージが表示される` |
| 操作可否 | `Then 「〇〇」ボタンがクリック可能である` / `クリック不可である` |

### And（追加条件）
Given/When/Then いずれの後にも使用可。同種の条件を追加する場合に使う。

## テストケースIDの命名規則

`TC-[機能略称]-[3桁連番]`

| 略称 | 対象 |
|------|------|
| PM | Permission Matrix（権限） |
| BL | Billing（請求・課金） |
| UI | UI/UX |
| DI | Data Integrity（データ整合性） |
| SC | Security（セキュリティ） |
| SN | Scenario（シナリオ） |

機能略称はプロジェクトごとに追加可。

### 詳細テストケースの略称

全体TCと詳細TCでIDが重複しないよう、詳細ファイルでは末尾に `D` を付与する。

| 略称 | 対象 | 用途 |
|------|------|------|
| PMD | Permission Matrix Detail（権限詳細） | 権限マトリクス詳細TC |
| BLD | Billing Detail（課金詳細） | 金額計算詳細TC |

例:
- 全体TC: `TC-PM-001`（概要レベル）
- 権限マトリクス詳細TC: `TC-PMD-001`（詳細レベル、単独実行可能）
- 金額計算詳細TC: `TC-BLD-001`（詳細レベル、単独実行可能）

## Scenario（テスト項目名）の書き方

自然な日本語で、**何が起きるか**を1行40字以内で要約する。

良い例:
- `Scenario: 経理担当者が請求画面にアクセスできる`
- `Scenario: 無料枠内の利用で請求額が0円になる`
- `Scenario: 権限なしユーザーがURL直接入力でアクセスを拒否される`

悪い例:
- `Scenario: 正しく動作する`（曖昧）
- `Scenario: テスト`（情報なし）
- `Scenario: CORPORATE_BILLING_MANAGERでサインインして/billingにアクセスすると請求・支払い画面が表示され支払方法の管理が可能`（長すぎ。これはGiven/When/Thenに分解する）

---

## 文章表現ルール

- 曖昧語禁止: 「正しく」「適切に」「問題なく」は使わない
- 一文は40字以内を目安に
- 操作対象のUI要素は「」で囲む（例: 「保存」ボタン、「メールアドレス」欄）
- URLパスは `/` から始める（例: /billing, /account）
- ロール名・ステータス名は英語表記のまま使う（例: CORPORATE_MEMBER, DRAFT）
- 金額は3桁カンマ区切りで単位を付ける（例: 3,000円）

## 優先度定義

| 優先度 | 定義 |
|--------|------|
| P1 | リリースブロッカー。必ずリリース前に実施 |
| P2 | 主要フロー。原則リリース前に実施 |
| P3 | エッジケース。リソースに応じて実施 |

- 金額計算に関するテストは原則 **P1** とする

- テスト観点: `YYYY-MM-DD_[機能名]_観点_v[N].md`
- テストケース: `YYYY-MM-DD_[機能名]_TC_v[N].md`

## スプレッドシート出力フォーマット（TSV）

テストケース・テスト観点をGoogleスプレッドシートで管理するためのTSV出力ルール。
Markdownドキュメントと**常にセットで**TSVファイルも生成する。

### 出力ファイル

テストケースのMarkdownを生成・更新した際は、以下のTSVも必ず生成する:

| ファイル | 命名規則 |
|---------|---------|
| テストケースTSV | `YYYY-MM-DD_[機能名]_全TC_スプレッドシート_v[N].tsv` |
| テスト観点TSV | `YYYY-MM-DD_[機能名]_テスト観点_スプレッドシート_v[N].tsv` |
| セットアップ手順 | `YYYY-MM-DD_[機能名]_スプレッドシート_セットアップ手順.md`（初回のみ） |

### テストケースTSVの列構成（15列）

```
TC-ID | 優先度 | タグ | 機能カテゴリ | シナリオ | 前提条件（Given） | 操作手順（When） | 期待結果（Then） | テスト結果 | 実行日 | 実行者 | 備考 | 対応観点ID | ソース | バグチケット
```

- **バグチケット**列: テスト実行で発見したバグのJIRAチケットURLを記録する。複数ある場合はカンマ区切りで列挙する。バグなしの場合は空欄とする。

### テスト観点TSVの列構成（12列）

```
観点ID | 分類 | 観点名 | サブ項目 | 確認内容 | 備考 | 判定 | 対象外理由 | 確認状況 | 確認者 | 関連TC | 対応TC数
```

- **対応TC数**列には `=SUMPRODUCT(--ISNUMBER(SEARCH(A行,TC!$M$2:$M$最終行)))` の数式を入れる（カンマ区切り複数観点ID対応）
- シートのタブ名は **「TC」** と **「観点」** を前提とする

### TSV生成時の必須ルール（列ズレ防止）

TSV生成時に列数の不一致を起こさないため、以下のルールを厳守する:

1. **全列を必ず出力する**: 値が空の列（テスト結果、実行日、実行者、バグチケット）も必ずタブ区切りで空フィールドとして出力する
2. **ヘッダーとデータ行の列数を一致させる**: ヘッダー行が15列ならデータ行も必ず15列（タブ文字14個）
3. **生成後に列数を検証する**: 全行のタブ文字数がヘッダー行と同一であることを確認する

**具体的な出力イメージ（15列の場合）:**
```
TC-AU-001\tP1\t@認証\t認証\tシナリオ文\t前提条件\t操作手順\t期待結果\t\t\t\t備考\t観点ID\tソース\t
```
↑ テスト結果・実行日・実行者・バグチケットが空でも `\t\t\t\t` として4つのタブを必ず出力する

**過去の不具合事例:**
- 空列（実行者）を省略して12列で出力した結果、備考→実行者列、ソース→備考列にズレが発生
- Pythonスクリプト等で生成する場合は `"\t".join([field1, field2, ..., field15])` のように全フィールドをリストに含めること
- Claude Code の Write ツールでTSVを生成した際、**行末の空列（バグチケット等）の末尾タブがトリムされ**、データ行が14列になった（ヘッダーは15列）。対策: TSV生成後に必ず `awk -F'\t' '{print NR": "NF" columns"}'` で列数を検証し、不一致があれば `awk -F'\t' 'NR==1{print $0} NR>1{print $0"\t"}'` で末尾タブを補完すること

### When/Then の記述ルール（TSV版）

TSVのWhen/Then列は**人間が手動テストで使うことを前提**に、可読性の高い文面にする。
Gherkin構文（Given/When/Then キーワード）は使わず、自然な日本語の手順・確認事項として書く。

**操作手順（When）の書き方:**
- 番号付きの手順にする: `1) ○○を開く → 2) △△を入力 → 3) 「送信」ボタンを押す`
- UI要素は「」で囲む
- パスは具体的に記載: `/billing にアクセスする`

**期待結果（Then）の書き方:**
- 「〜であること」「〜が表示されていること」の確認形式で記述
- 具体的な値を含める: `ステータスが「有効」と表示されていること`
- 複数の確認項目がある場合は箇条書き風に列挙

**悪い例（TSVには使わない）:**
```
When CORPORATE_OWNERでサインインして/billingにアクセスする
Then 請求画面が正しく表示される
```

**良い例:**
```
操作手順: 1) CORPORATE_OWNERでサインインする → 2) サイドメニューの「請求・支払い」をクリック
期待結果: 請求・支払い画面（/billing）に遷移し、今月の請求サマリーと支払方法一覧が表示されていること
```

### クロスリファレンス（双方向参照）

テストケースとテスト観点は必ず相互参照可能にする:

1. **TCシート → 観点シート**: M列（対応観点ID）に、そのTCが検証する観点IDをカンマ区切りで記録（複数可）
2. **観点シート → TCシート**: K列（関連TC）に、その観点をカバーするTC-IDをカンマ区切りで記録
3. **観点シート 対応TC数**: L列にSUMPRODUCT+SEARCH数式で自動カウント（カンマ区切り対応）
4. **双方向リンク**: Googleスプレッドシートインポート後、Apps Scriptで観点ID/TC-IDをクリック可能なリンクに変換

### TC→観点マッピングの原則

- 各TCには**複数の観点ID**をカンマ区切りで割り当て可能（1:N）
- 観点側の関連TCも複数可（N:M）
- 1つのTCが複数の観点を検証する場合は全ての関連観点IDを列挙する
- マッピングは機能カテゴリ・テスト目的から判断:
  - 正常系操作のTC → 仕様正常系の観点
  - エラー系のTC → 仕様異常系の観点
  - 権限テスト → ロール別権限の観点
  - 金額計算 → 該当する計算ロジックの観点
- 「対象外」判定の観点（テスト不要と判断されたもの）にはTCをマッピングしない

## 具体例

以下は権限テストケースの記述例。

```markdown
## Feature: 権限マトリクス（画面アクセス）

### TC-PM-001 @P1 @権限

**Scenario**: オーナーがアカウント管理画面にアクセスできる

  Given PERSONAL_OWNERでサインインしている
  When /account にアクセスする
  Then アカウント管理画面が表示される
    And 「プロファイル編集」ボタンがクリック可能である

**結果**: -
**備考**: -

---

### TC-PM-007 @P1 @権限

**Scenario**: メンバーが請求画面へのアクセスを拒否される

  Given PERSONAL_MEMBERでサインインしている
  When /billing にアクセスする
  Then 権限エラー画面が表示される
    And サイドメニューに「請求・支払い」が表示されない

**結果**: -
**備考**: -

---

### TC-BL-004 @P1 @金額計算

**Scenario**: 個人プラン少量利用でminimumFeeが適用される

  Given PERSONAL契約の組織でサインインしている
    And 月間利用額が600円を超え3,000円未満である
  When /billing にアクセスする
    And 今月の請求書を開く
  Then freeTierAmount が 600円 である
    And taxableAmount が 3,000円 である
    And taxAmount が floor(3,000 × 0.10) = 300円 である
    And total が 3,300円 である

**結果**: -
**備考**: minimumFee適用パターン
```
