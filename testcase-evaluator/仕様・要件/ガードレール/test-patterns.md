## よく使うテストパターン

### パターン1: CRUD基本フロー
```
前提条件 → 新規作成 → 一覧に反映確認 → 詳細確認 → 編集 → 削除 → 一覧から消えたか確認
```
- 一覧に何もない状態 → 新規作成 → 一覧に反映確認 → 詳細確認 → 編集 → 削除 → 一覧から消えたか確認
- 一覧に1件ある状態 → 新規作成 → 一覧に反映確認 → 詳細確認 → 編集 → 削除 → 一覧から消えたか確認
- 一覧に複数件ある状態 → 新規作成 → 一覧に反映確認 → 詳細確認 → 編集 → 複数削除 → 一覧から消えたか確認
- 削除は、詳細画面から削除を選択するパターンと、一覧上で削除したいものを全て選択してから一括削除するやり方があるので混ぜる

### パターン2: 権限マトリクス
```
機能 × ロール の組み合わせ全網羅
（管理者・一般ユーザー・ゲスト など）
```
- 権限:一般、管理者、集計者、取引先管理者、承認者。権限は複数同時に持つことができるので、単一権限、複数権限全パターンがありうる。
- 制限:集計画面は集計者権限がないと表示されない。システム設定は管理者または集計者権限がないと表示されない。承認画面は承認者権限がないと表示されない。
- 特殊ロール:代理人。代理人は、指定のメンバーが行う申請、承認行為を代理で実行できる。
- URL直接アクセス:メニューに表示されない画面でも、URLを直接入力してアクセスできてしまわないか確認する。メニュー非表示＝アクセス不可 ではない点に注意。

### パターン3: エラー回復フロー
```
エラー発生 → エラーメッセージ確認 → 再操作 → 正常完了
```
- エラー発生 → エラーメッセージ確認 → 再操作 → 正常完了　※リトライで解消するもののみ
- エラー発生 → エラーメッセージ確認　※リトライで解消しないもの。エラーメッセージによって理由が示されていることが期待値

### パターン4: 金額計算テスト
```
入力経路別 × 数値パターン × 丸め処理 × 税率設定 の組み合わせ
```
- 入力経路: 手入力、自動入力（OCR等）、自動仕訳、定期支払、カード連携など、同じ金額欄でも入力される経路が複数ある
- 数値パターン: 整数、小数、負の値、0 をそれぞれ試す
- 丸め処理: 切捨て・切上げ・四捨五入が発生するケースと発生しないケース（割り切れる/割り切れない）の両方を含める
- 按分: 割合指定で按分する場合、合計が元の金額と一致するかを確認（端数の処理方法に注意）
- 税率: 消費税、源泉徴収税などの自動計算。税抜/税込の設定切替で金額が正しく変動するか

### パターン5: 時間境界テスト
```
境界タイミング直前の操作 → 時間経過 → 境界タイミング直後の状態確認
```
- 日跨ぎ（23:59→0:00）、月跨ぎ（月末→翌月1日）、年跨ぎ（12/31→1/1）、年度跨ぎ（3/31→4/1）
- ステータスの自動遷移（有効期限切れ、バッチ処理のトリガー等）が正しいタイミングで発火するか
- 長期間放置しても機能が維持されるか（セッション、データの有効期限）
- テスト環境で時刻を操作する場合の手順を事前に確認しておく

### パターン6: リリース前後比較テスト
```
リリース前の状態を記録 → リリース実施 → リリース後に同じ操作・データで確認
```
- リリース前に作成済みの設定・データが、リリース後もそのまま参照・操作できるか
- 設定値が勝手に変わっていないか（削除される、不要な設定が増える等）
- 途中状態のデータ（申請中・承認待ち等）がリリースを跨いでも正常に継続処理できるか
- ロールバック時に元の状態に復帰できるか

## テストケース設計の注意点

### 事前条件の記載
- テスト実行前に必要なデータ状態を必ず明記する
- 他テストとのデータ干渉に注意（テストの独立性）
- 準備の手間を削減するため、同じ準備から開始できるテストケースは並べて記載すること

### 期待結果の書き方
- 曖昧な表現NG: 「正しく表示される」※期待値が感覚的な場合(UXなど)はこの限りではない
- 具体的に書く: 「〇〇というメッセージが赤字で表示される」

### 優先度の付け方
- P1（必須）: リリースブロッカーになる機能
- P2（重要）: 主要ユーザーフローに関わる
- P3（通常）: 補助的な機能・エッジケース
- 金額計算に関するテストは原則 **P1** とする（小数点の丸め・按分を含むものは特に）

## チーム固有のナレッジ
<!-- 経験から学んだことを随時追記してください -->

### ハマりやすいポイント
- マスタデータ（部署、プロジェクト等）を更新・削除した際、そのマスタを参照している過去の経費・請求書データに異常が出ることがある
- 経費・請求書に入力フィールドが追加された場合、検索画面・集計画面・会計データ定期出力の設定にも波及する
- アプリ版（iOS/Android）の影響確認を忘れやすい。経費の作成・申請・承認、支払依頼申請の承認、カード連携はアプリも影響範囲
- 小数点が発生しうる金額計算（切捨て・切上げ、割合按分）はバグの発生頻度が高く、発生時の緊急度も高い

### 効率的なテスト実行順序
- 実行に時間がかかるものから実施する
- 不具合が発生した場合に影響範囲が広いものから実施する
