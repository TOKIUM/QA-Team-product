# テスト設計ワークフロー（標準手順）

## 概要

テスト設計の依頼を受けた際に、以下の5フェーズを**途中の確認を挟まず最後まで自動実行**する。
ユーザーへの質問・確認は原則として作業開始前のみ。開始後は全フェーズを連続で完了させる。

---

## 前提条件（作業開始前に確認すること）

以下の情報がそろっていない場合のみ、作業開始前にユーザーに確認する:

1. **テスト対象**: どのプロダクト・機能が対象か
2. **ソースの場所**: ソースコード・ドキュメントのパス
3. **既存テスト設計物**: 既にテスト観点やTCがある場合はそのパス（新規の場合はなし）

---

## Phase 0: フォルダ準備（新規テスト対象の場合）

新規テスト対象のテスト設計を開始する場合、以下のフォルダを作成する:

```
output/[テスト対象名]/
  drafts/
  reviewed/
  versions/
```

既存テスト対象への増分更新の場合は既存フォルダを使用する。

---

## Phase 1: テスト観点作成

### インプット
- ソースコード（リポジトリ全体）
- 仕様ドキュメント（docs/フォルダ等）
- `knowledge-base/qa-perspectives.md`（観点チェックリスト）

### 処理
1. `qa-perspectives.md` を読み込む
2. ソースコードのディレクトリ構造を把握する
3. 主要ファイル（ルーティング、ミドルウェア、DB スキーマ、ビジネスロジック）を読む
4. `qa-perspectives.md` の各チェック項目を順に評価し、対象/対象外を判定する
5. 対象外の場合は理由を明記する
6. テスト観点ドキュメントを `style-guide.md` のフォーマットに従って生成する

### アウトプット
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_観点_v1.md`

---

## Phase 2: テストケース作成

### インプット
- Phase 1 のテスト観点ドキュメント
- ソースコード（Phase 1 で読んだもの）
- `knowledge-base/style-guide.md`（Gherkin記述ルール）
- `knowledge-base/test-patterns.md`（テストパターン）

### 処理
1. テスト観点の各項目に対してテストケースを作成する
2. 優先度（P1/P2/P3）を設定する
3. TC-IDを命名規則に従って付与する
4. Gherkin形式（Given/When/Then）で記述する
5. 観点カバレッジ: 全ての「対象」判定の観点に最低1つのTCを紐付ける
6. 高リスク領域（金額計算、権限制御等）は詳細TCファイルを別途作成する

### アウトプット
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_TC_v1.md`
- （必要に応じて）`output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_[詳細領域]_TC_v1.md`

---

## Phase 3: ドキュメント突き合わせ（ギャップ分析）

### インプット
- Phase 1-2 で作成した全テスト設計物
- ソースの仕様ドキュメント（docs/フォルダ全体）

### 処理
1. 仕様ドキュメントを全ファイル読み込む
2. ドキュメントに記載のある機能・仕様・ルール・制約を網羅的にリストアップする
3. 既存のテスト観点・TCと突き合わせ、カバーされていない項目を特定する
4. ギャップを優先度別（P1/P2/P3）に整理する

### 判定基準
- **ギャップあり**: Phase 4 へ進む
- **ギャップなし**: Phase 4 をスキップして Phase 5 へ進む

---

## Phase 4: 加筆修正（ギャップ補完）

### インプット
- Phase 3 のギャップ分析結果
- 既存テスト設計物

### 処理
1. ギャップに対応する新規TCを作成する（新バージョンのTCファイル）
2. 必要に応じて新規テスト観点を追加する
3. 既存の観点⇔TC双方向マッピングを更新する
4. バージョンは既存ファイルの次バージョンとして作成（v1→v2→v3）

### アウトプット
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_TC_v[N].md`（追加TC）
- テスト観点は元ファイルのバージョンアップ、またはTSVで反映

---

## Phase 5: TSV生成（スプレッドシート用）

### インプット
- 全てのテスト観点ドキュメント
- 全てのテストケースドキュメント

### 処理
1. **TC TSV** を生成する:
   - `style-guide.md` の15列フォーマットに従う
   - 全TCを統合（全体TC + 詳細TC）
   - TC→観点マッピング（M列: 対応観点ID）をカンマ区切りで設定
   - 空列（テスト結果、実行日、実行者）も必ず出力する（列ズレ防止）
   - 全行の列数がヘッダーと一致することを検証する
2. **観点TSV** を生成する:
   - `style-guide.md` の12列フォーマットに従う
   - 観点→TCマッピング（K列: 関連TC）をカンマ区切りで設定
   - 対応TC数列にSUMPRODUCT数式を設定する
   - 全行の列数がヘッダーと一致することを検証する
3. **セットアップ手順** を生成する（初回のみ、または大幅変更時）:
   - Googleスプレッドシートへのインポート手順
   - Apps Script（双方向リンク設定スクリプト）
   - 動作確認チェックリスト

### アウトプット
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_全TC_スプレッドシート_v[N].tsv`
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_テスト観点_スプレッドシート_v[N].tsv`
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_スプレッドシート_セットアップ手順.md`（初回のみ）

---

## Phase 6: 用語集作成（専門用語マッピング）

### 目的

テストケース内の専門用語（変数名、内部コード、APIパラメータ名など）は、テスト実行者にとって理解しにくい。
画面上の表示名や操作手順との対応を明確にし、テスト実行の正確性・効率を向上させる。

### インプット
- 全てのテストケースドキュメント（Phase 2〜4 の成果物）
- ソースコード（UI定義、ロール定義、Enum定義、i18nファイル等）
- 仕様ドキュメント

### 処理

1. **専門用語の抽出**: テストケース全体を走査し、一般常識では理解できない用語を網羅的にリストアップする
   - 判定基準: 英大文字スネークケース（例: `PERSONAL_OWNER`）、キャメルケース変数名、内部コード、API固有のパラメータ名など
   - 略語・頭字語（JWT、SSO等）も対象とする
2. **画面表示名のマッピング**: 抽出した各用語について、ソースコード（UI定義、i18nファイル等）を調査し、画面上でどのように表示されているかを特定する
   - 用語 → 画面表示名の対応表を作成する
   - 画面上に表示されない内部用語の場合はその旨を明記する
3. **状態・パターンの準備方法の調査**: 各用語が表す状態・ロール・設定を、テスト環境上でどのように準備するかを調査する
   - 例: 権限名 → その権限を持つユーザーの作成手順
   - 例: ステータス名 → そのステータスに遷移させる操作手順
   - 必要に応じて状態遷移図やフローチャート（Mermaid記法）を作成する
4. **用語集ドキュメントの作成**: 手順1〜3の情報を以下の構成でまとめる
   - カテゴリ別の用語対応表（ロール系、ステータス系、機能系、認証系など）
   - 状態遷移図・フローチャート（該当する場合）
   - テスト環境での準備手順
5. **セルフチェック**: 作成した用語集の正確性を検証する
   - ソースコードの定義と矛盾がないか
   - マッピング漏れがないか（TCに登場する全用語がカバーされているか）
   - 準備手順が実行可能か
6. **最終版を出力する**

### アウトプット
- `output/[テスト対象名]/drafts/YYYY-MM-DD_[機能名]_用語集_v[N].md`

### 用語集ドキュメントの構成

```
# [機能名] テスト用語集

## メタ情報
- 対象機能:
- 作成日:
- 対応テストケース:
- ステータス: Draft / Reviewed / Approved

## 1. 用語対応表

### 1-1. ロール・権限
| 内部名称 | 画面表示名 | 説明 | 準備方法 |
|----------|-----------|------|----------|

### 1-2. ステータス・状態
| 内部名称 | 画面表示名 | 説明 | 遷移条件 |
|----------|-----------|------|----------|

### 1-3. 認証・セッション
| 内部名称 | 画面表示名 | 説明 | 備考 |
|----------|-----------|------|------|

### 1-4. その他（機能固有）
| 内部名称 | 画面表示名 | 説明 | 備考 |
|----------|-----------|------|------|

## 2. 状態遷移図・フローチャート
（Mermaid記法で記述。該当するものがある場合のみ）

## 3. テスト環境での準備手順
（各状態・パターンの具体的な作成手順）
```

全フェーズ完了後、以下のサマリーを出力する:

```

|---|---------|------|
| 1 | ... | ... |

### 数値サマリー
| 項目 | 件数 |
|------|------|
| テストケース合計 | N件 |
| テスト観点合計 | N件 |
| P1 TC | N件 |
| P2 TC | N件 |
| P3 TC | N件 |

### 次のステップ
- TSVをGoogleスプレッドシートにインポート
- Apps Scriptでリンク設定を実行
- 用語集の内容をチームでレビュー
- レビュアーによるレビュー
```

---

## 増分更新（既存テスト設計物がある場合）

既にテスト設計物が存在する状態で追加の依頼を受けた場合:

1. 既存の観点・TCを全て読み込む
2. 新たな要求に対して Phase 1-4 を差分として実行する
3. Phase 5 では既存TSVに新規行を追加した新バージョンを生成する
4. 既存の観点⇔TC双方向マッピングも全て更新する
5. SUMPRODUCT数式のレンジも新しい行数に合わせて更新する

---

## 注意事項

- **途中で止めない**: Phase 1 から Phase 6 まで、ユーザーへの確認を挟まず一気通貫で実行する
- **列ズレ防止**: TSV生成時は必ず列数を検証する（`style-guide.md` のTSV生成ルール参照）
- **バージョン管理**: 既存ファイルは上書きせず、必ず新バージョンとして作成する
- **双方向マッピング**: TC→観点、観点→TCの両方向を必ず設定する
- **検証**: TSV生成後に全行の列数一致を自動チェックする
