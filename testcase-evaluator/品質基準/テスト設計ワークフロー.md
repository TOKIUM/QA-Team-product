# テスト設計ワークフロー（標準手順）

## 概要

テスト設計の依頼を受けた際に、以下の5フェーズを**途中の確認を挟まず最後まで自動実行**する。
ユーザーへの質問・確認は原則として作業開始前のみ。開始後は全フェーズを連続で完了させる。

---

## 前提条件（作業開始前に確認すること）

以下の情報がそろっていない場合のみ、作業開始前にユーザーに確認する:

1. **テスト対象**: どのプロダクト・機能が対象か
2. **ソースの場所**: 仕様書・JIRAチケット・ソースコードのパス
3. **既存テスト設計物**: 既にテスト観点やTCがある場合はそのパス（新規の場合はなし）

---

## Phase 1: テスト観点作成

### インプット
- 仕様書 / JIRAチケット / ソースコード
- `品質基準/観点チェックリスト.md`（16機能+7非機能観点）
- `品質基準/プロダクト情報/`（プロダクト固有の画面構成・テスト観点）

### 処理
1. 観点チェックリストを読み込む
2. 仕様書・ソースを分析し、各チェック項目を順に「対象/対象外」判定する
3. 対象外の場合は理由を明記する
4. プロダクト情報から固有の重点観点・対象外になりやすい観点を参照する
5. テスト観点をExcelテンプレートの観点シートに反映する

### アウトプット
- Excelテンプレートの観点シート（環境/OS/ブラウザ/主要観点/権限を自動選択）

---

## Phase 2: テストケース作成

### インプット
- Phase 1 のテスト観点
- 仕様書・ソースコード
- `品質基準/記述ルール.md`（記述品質基準）
- `品質基準/テストパターン.md`（CRUD/権限/金額計算/時間境界等6パターン）

### 処理
1. テスト観点の各項目に対してテストケースを作成する
2. 優先度（P1/P2/P3）を設定する（金額計算は必ずP1）
3. 記述ルールに従って記述する（曖昧語禁止、「〜こと」語尾、1期待値1こと）
4. テストパターンを参照し、正常系・異常系・境界値・エッジケースを網羅する
5. 観点カバレッジ: 全ての「対象」判定の観点に最低1つのTCを紐付ける
6. 高リスク領域（金額計算、権限制御等）は詳細TCを追加する

### アウトプット
- JSON形式のテストケース → `generate_testcases.py` でExcel変換

---

## Phase 3: ドキュメント突き合わせ（ギャップ分析）

### インプット
- Phase 1-2 で作成した全テスト設計物
- 仕様書全体

### 処理
1. 仕様書に記載のある機能・仕様・ルール・制約を網羅的にリストアップする
2. 既存のテスト観点・TCと突き合わせ、カバーされていない項目を特定する
3. ギャップを優先度別（P1/P2/P3）に整理する
4. 影響範囲チェックリスト（5-1〜5-6）を確認する

### 判定基準
- **ギャップあり**: Phase 4 へ進む
- **ギャップなし**: Phase 4 をスキップして Phase 5 へ進む

---

## Phase 4: 加筆修正（ギャップ補完）

### 処理
1. ギャップに対応する新規TCを作成する
2. 必要に応じて新規テスト観点を追加する
3. 観点⇔TC双方向マッピングを更新する

### アウトプット
- 追加テストケースJSON → Excel再変換

---

## Phase 5: 自動評価 + 用語集（Evaluator-Optimizerループ）

### 処理
1. `evaluate_testcases.py` で自動評価（100点満点）
2. `evaluate_perspectives.py` で観点評価（100点満点）
3. **80点以上**: 完了
4. **80点未満**: 減点箇所を修正して再評価（最大3回反復）
5. 専門用語が多い場合は用語集を作成する

### 用語集の構成（必要な場合のみ）
- ロール・権限: 内部名称→画面表示名→準備方法
- ステータス・状態: 内部名称→画面表示名→遷移条件
- 認証・セッション: 用語→説明→備考
- 状態遷移図（Mermaid記法、該当する場合のみ）

---

## 完了時の出力

全フェーズ完了後、以下のサマリーを出力する:

```
## 完了サマリー

### 生成ファイル一覧
| # | ファイル | 内容 |
|---|---------|------|
| 1 | ... | ... |

### 数値サマリー
| 項目 | 件数 |
|------|------|
| テストケース合計 | N件 |
| テスト観点合計 | N件 |
| P1 TC | N件 |
| P2 TC | N件 |
| P3 TC | N件 |
| 評価スコア | N点 |
```

---

## 増分更新（既存テスト設計物がある場合）

1. 既存の観点・TCを読み込む
2. 新たな要求に対して Phase 1-4 を差分として実行する
3. 既存Excelに新規TCを追加した新バージョンを生成する
4. 観点⇔TC双方向マッピングも更新する

---

## 注意事項

- **途中で止めない**: Phase 1 から Phase 5 まで一気通貫で実行する
- **出力形式**: Excelテンプレート準拠が最優先（Gherkin/TSVは不採用）
- **金額計算は必ずP1**: 小数点丸め・按分は特に注意
- **曖昧語禁止**: 「正しく」「適切に」「問題なく」→ 具体的状態を記述
- **プロダクト固有情報**: `品質基準/プロダクト情報/` を必ず参照する
