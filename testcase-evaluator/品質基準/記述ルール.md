# テストケース記述ルール

テストケース・テスト観点の記述に関する統一基準です。

---

## 1. 基本原則

| ルール | 詳細 |
|--------|------|
| **曖昧語禁止** | 「正しく」「適切に」「問題なく」「正常に」「仕様通りに」は使わない |
| **1ケース1期待値** | 期待値セルに「こと」は原則1回まで。複数ある場合はケースを分割 |
| **我流単語禁止** | 仕様書・PRDに記載の正式名称を使う。略語・造語は不可 |
| **手順は省略しない** | 同カテゴリの先頭ケースに手順を書き後続は省略可（グループ共有手順） |
| **UI要素は「」で囲む** | 例: 「保存」ボタン、「メールアドレス」欄 |
| **金額は必ずP1** | 金額計算（小数点丸め・按分含む）のテストケースは優先度P1必須 |

---

## 2. テスト手順のフォーマット

手順セルは以下の統一フォーマットで記述する（`common.py` の `format_test_steps()` で自動整形）。

### 正しいフォーマット

```
【前提条件】
・パスワード変更画面でパスワードを入力済みであること
・新しいパスワードが現在のパスワードと異なること

【実行手順】
1. 「パスワード(再入力)」欄に新しいパスワードと同一の値を入力する
2. 「パスワードを変更する」ボタンをクリックする
```

### ルール

- **前提条件**がある場合は `【前提条件】` セクションを先頭に置く
- 各前提条件は `・` で始める
- 前提条件と実行手順の間に空行を入れる
- **実行手順**は `【実行手順】` セクション内に番号付きで記載
- 前提条件がない場合は `【実行手順】` から開始する

### 悪い例（NGフォーマット）

```
【前提条件】パスワード変更画面でパスワードを入力済みであること 1.「パスワード(再入力)」欄に...
```

改行なし・セクション区切りなしは視認性が低く、評価で減点される。

## 3. 期待値の書き方

### 良い例

| 書き方 | 例 |
|--------|-----|
| 具体的な表示内容 | 「パスワードを変更しました」というメッセージが表示されること |
| 画面遷移 | ログイン画面（/login）に遷移すること |
| 状態変化 | ステータスが「有効」から「無効」に変更されること |
| 数値確認 | 合計金額が 3,300円（税込）と表示されること |
| 非表示確認 | 「削除」ボタンが表示されないこと |

### 悪い例（NG）

| NG表現 | 改善案 |
|--------|--------|
| 正常に動作すること | ログイン後画面に遷移すること |
| 問題ないこと | エラーメッセージが表示されないこと |
| 仕様通りであること | [具体的な仕様内容]が表示されること |
| 適切に表示されること | 「〇〇」が赤字で表示されること |
| 正しく計算されること | 合計が XX円と表示されること |

### 複数期待値の分割例

修正前（NG）:
> 表示されること（各要素が表示されていること）

修正後（OK）:
> ログイン画面にメール欄・パスワード欄・ログインボタンが表示されること

---

## 4. 優先度の定義

| 優先度 | 定義 | 例 |
|--------|------|-----|
| P1 | リリースブロッカー。必ずリリース前に実施 | 金額計算、主要認証フロー |
| P2 | 主要フロー。原則リリース前に実施 | 通常操作の正常系・異常系 |
| P3 | エッジケース。リソースに応じて実施 | 補助機能、互換性確認 |

**必須ルール**: 金額計算に関するテストは原則 **P1** とする（小数点の丸め・按分を含むものは特に）

---

## 5. 環境列の書き方

- 対象環境の列に `○` を記入、対象外は空欄
- 全環境対象の場合でも各列に明示的に `○` を記入する
- テンプレートに「未」等のデフォルト値がある場合は必ずクリアする

---

## 6. 観点シートの書き方

- チェックリスト形式を維持する
- 各観点について「対象」「対象外」を明記し、対象外の場合は理由を必ず記載
- 観点とテストケースは双方向で紐づける（観点→TC、TC→観点）
