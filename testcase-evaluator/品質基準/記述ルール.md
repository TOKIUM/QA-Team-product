# テストケース記述ルール

テストケース・テスト観点の記述に関する統一基準です。

---

## 1. 基本原則

| ルール | 詳細 |
|--------|------|
| **曖昧語禁止** | 「正しく」「適切に」「問題なく」「正常に」「仕様通りに」は使わない |
| **1ケース1期待値** | 期待値セルに「こと」は原則1回まで。複数ある場合はケースを分割 |
| **我流単語禁止** | 仕様書・PRDに記載の正式名称を使う。略語・造語は不可 |
| **手順は省略しない** | 同カテゴリの先頭ケースに手順を書き後続は省略可（グループ共有手順） |
| **UI要素は「」で囲む** | 例: 「保存」ボタン、「メールアドレス」欄 |
| **金額は必ずP1** | 金額計算（小数点丸め・按分含む）のテストケースは優先度P1必須 |

---

## 2. テスト手順のフォーマット

手順セルは以下の統一フォーマットで記述する（`common.py` の `format_test_steps()` で自動整形）。

### 正しいフォーマット

```
【前提条件】
・パスワード変更画面でパスワードを入力済みであること
・新しいパスワードが現在のパスワードと異なること

【実行手順】
1. 「パスワード(再入力)」欄に新しいパスワードと同一の値を入力する
2. 「パスワードを変更する」ボタンをクリックする
```

### ルール

- **前提条件**がある場合は `【前提条件】` セクションを先頭に置く
- 各前提条件は `・` で始める
- 前提条件と実行手順の間に空行を入れる
- **実行手順**は `【実行手順】` セクション内に番号付きで記載
- 前提条件がない場合は `【実行手順】` から開始する

### 悪い例（NGフォーマット）

```
【前提条件】パスワード変更画面でパスワードを入力済みであること 1.「パスワード(再入力)」欄に...
```

改行なし・セクション区切りなしは視認性が低く、評価で減点される。

## 3. 期待値の書き方

### 良い例

| 書き方 | 例 |
|--------|-----|
| 具体的な表示内容 | 「パスワードを変更しました」というメッセージが表示されること |
| 画面遷移 | ログイン画面（/login）に遷移すること |
| 状態変化 | ステータスが「有効」から「無効」に変更されること |
| 数値確認 | 合計金額が 3,300円（税込）と表示されること |
| 非表示確認 | 「削除」ボタンが表示されないこと |

### 悪い例（NG）

| NG表現 | 改善案 |
|--------|--------|
| 正常に動作すること | ログイン後画面に遷移すること |
| 問題ないこと | エラーメッセージが表示されないこと |
| 仕様通りであること | [具体的な仕様内容]が表示されること |
| 適切に表示されること | 「〇〇」が赤字で表示されること |
| 正しく計算されること | 合計が XX円と表示されること |

### 複数期待値の分割例

修正前（NG）:
> 表示されること（各要素が表示されていること）

修正後（OK）:
> ログイン画面にメール欄・パスワード欄・ログインボタンが表示されること

---

## 4. 優先度の定義

| 優先度 | 定義 | 例 |
|--------|------|-----|
| P1 | リリースブロッカー。必ずリリース前に実施 | 金額計算、主要認証フロー |
| P2 | 主要フロー。原則リリース前に実施 | 通常操作の正常系・異常系 |
| P3 | エッジケース。リソースに応じて実施 | 補助機能、互換性確認 |

**必須ルール**: 金額計算に関するテストは原則 **P1** とする（小数点の丸め・按分を含むものは特に）

---

## 5. 環境列の書き方

- 対象環境の列に `○` を記入、対象外は空欄
- 全環境対象の場合でも各列に明示的に `○` を記入する
- テンプレートに「未」等のデフォルト値がある場合は必ずクリアする

---

## 6. 観点シートの書き方

- チェックリスト形式を維持する
- 各観点について「対象」「対象外」を明記し、対象外の場合は理由を必ず記載
- 観点とテストケースは双方向で紐づける（観点→TC、TC→観点）
- **階層構造で記述する**（5段階）:
  - 確認すること① → 確認すること② → 画面 → 確認対象 → 詳細①（→詳細②）
  - 例: 「既存機能への影響」→「パスワード再設定」→「ログイン画面」→「パスワード変更後のログイン」→「変更前のPWでログインできないか」
- **画面名はパンくず形式**: 「TH 請求書一覧 > 請求書詳細 添付ファイルタブ」のように画面遷移パスを明示
- **目次を設ける**: 複数目次がある場合は(目次1)TH (目次2)WDL のように分類し、形式（リスト形式/デシジョンテーブル形式）を明記

---

## 7. デシジョンテーブルの使用基準

以下の場合はリスト形式ではなく**デシジョンテーブル形式**で観点を記述する:

- **条件の掛け合わせが3条件以上**: 例: 制限フラグ×ログイン方式×ユーザー種別×サブドメイン×IP制限×SAML
- **条件の組み合わせで結果が変わる**: 例: フラグ=当日+メアドPW→拒否、フラグ=当日+SSO→許可

### デシジョンテーブルの構造

```
| 条件    | パターン1 | パターン2 | ... |
|---------|----------|----------|-----|
| 条件A   | 値1      | 値2      |     |
| 条件B   | 値1      | 値1      |     |
| 動作    | 結果A    | 結果B    |     |
```

- 行: 条件(上段) + 動作(下段)
- 列: パターン番号
- マルチテナント・境界値は別テーブルに分離

---

## 8. テストシナリオ（開発側連携）シート

複雑な前提条件の設定が必要なテストでは、**テストシナリオシート**を作成する:

- S1, S2, ... の番号付きシナリオ
- 各シナリオに以下を記載:
  - **使用ユーザー**: 区分（Xユーザー/Yユーザー等）
  - **前提条件**: 状態設定の説明
  - **セットアップ**: Railsコンソールコマンド等の具体的な手順
  - **操作**: テスト実行時の操作
  - **期待結果**: 判定基準
  - **テストケースとの紐づけ**: 「テストケース_目次1 no.4〜5」

---

## 9. 期待値の分割ルール

1つの操作に対して**確認ポイントが複数ある場合は別TCに分割**する:

| 操作 | 分割前（NG） | 分割後（OK） |
|------|-------------|-------------|
| ログイン | ログインが成功し、画面に遷移しエラーがないこと | TC1: ログインが成功すること / TC2: ログイン後画面に遷移すること / TC3: エラーメッセージが表示されないこと |

- **必須分割対象**: 画面遷移確認 + エラーメッセージ非表示確認 は常に独立TCに
- 先頭TCに手順を書き、後続TCは手順=「-」でグループ化する

---

## 10. 画面名の書き方

テストケース・観点シートの「画面」列は、テスターが迷わず特定できる名称で記載する。

### 3つの形式（参考例から抽出）

| 場面 | 形式 | 例 |
|------|------|-----|
| 単一画面のテスト | **シンプル名** | `ログイン画面`、`パスワード再設定画面` |
| 画面内の特定タブ/セクション | **パンくず `>`** | `TH 請求書一覧 > 請求書詳細 添付ファイルタブ` |
| 画面遷移を検証するテスト | **フロー `→`** | `THログイン画面→TOKIUM ID認証画面→THログイン後画面` |

### 必須ルール

- **URLパスを画面名に含めない**: `TH 取引先（/partners）` → `TH 取引先` （URLは手順や備考に記載）
- **UI実装用語を画面名に含めない**: `モーダル`、`ドロワー`、`コンポーネント` → 操作目線の名称を使う（例: `Web送付設定画面`）
- **プロダクトプレフィックス**: 複数プロダクトを跨ぐ場合は先頭に `TH` `WDL` 等を付ける
- **パンくず形式の使用場面**: リグレッションテストで既存画面への影響を確認する場合は必ずパンくず形式で画面パスを明示する

### NGパターン

| NG | 理由 | OK |
|----|------|-----|
| `TH 取引先（/partners）モーダル Web送付設定` | URL混入+実装用語 | `TH 取引先 > Web送付設定` |
| `TH 請求書詳細（/invoices/{UUID}）添付ファイルタブ` | URL混入 | `TH 請求書詳細 > 添付ファイルタブ` |
| `全画面共通` | テスターが特定不可 | 具体的な画面名を列挙 |
| `設定画面` | 曖昧すぎる | `TH 管理画面 > テナント設定` |

---

## 11. テスト実行列ヘッダー

- 観点シートの「環境」欄で選定した組み合わせを**そのままTC実行列ヘッダーに使用**する
- 例: 環境=TH単体〇 → ヘッダー=「TH単体」（Win11/Chrome, Win11/Edge, Win11/Firefox, Mac/Safari）
- 例: 環境=TK+TI+TD〇 → ヘッダー=「TK+TI+TD」
- 汎用的な「Win11/Chrome」等ではなく、テスト対象の組み合わせ名で表記する

---

## 12. グループ化TC（親子構造）

同一操作に対して複数の検証ポイントがある場合、**親TC + 子TC**でグループ化する。

### 構造

```
No.1: 確認すること="レスポンスヘッダ確認", 手順="【実行手順】1. ログイン画面を開く..."
No.2: 確認すること="",                    手順="-"
No.3: 確認すること="",                    手順="-"
```

### ルール

- **親TC**: 手順を記載し、「確認すること」に分類名を設定
- **子TC**: 手順=`-`（ハイフン）、「確認すること」=空欄。期待値のみ異なる
- **分割基準**: 1操作に対し「成功確認」「画面遷移確認」「エラー非表示確認」「データ永続化確認」等を別TCに
- **セクション9の期待値分割と併用**: 分割した期待値をグループ化TCとして構造化する

### JSON生成時の対応

```json
[
  {"confirm": "パスワード認証", "steps": "1. ログイン画面を...", "expected": "ログインが成功すること", "group_parent": true},
  {"confirm": "", "steps": "-", "expected": "ログイン後画面に遷移すること", "group_parent": false},
  {"confirm": "", "steps": "-", "expected": "エラーメッセージが表示されないこと", "group_parent": false}
]
```

## 13. 観点シートの記入密度基準

観点シートはテンプレートの枠を埋めるだけでなく、**テスターが実行判断できる密度**で記入する。

### 必須記入項目

| 項目 | 基準 | NG例 |
|------|------|------|
| 確認すること①〜② | 2段階以上の分類 | 「機能確認」のみで止める |
| 画面（L3） | パンくず形式の具体名 | 空欄 or 「全画面」 |
| 確認対象（L4） | UI要素名 or 操作対象 | 空欄 |
| 詳細①（L5） | 具体的な確認ポイント | 空欄 |
| 対象/対象外 | 全行に明記 | 空欄のまま放置 |
| 対象外理由 | 対象外の場合は必ず理由 | 「対象外」のみ |

### 密度の目安（参考例ベンチマーク）

- **L3（画面）記入率**: 80%以上（全行のうち画面名が具体的に記載されている割合）
- **L5（詳細）記入率**: 60%以上（L4まで記入した行のうち詳細まで記載されている割合）
- **対象外理由記入率**: 100%（対象外にした全行に理由あり）

### セルフチェック

観点シート完成時に以下を自問する:
1. テスターがこの観点シートだけ読んで「何を確認すればいいか」分かるか？
2. 各行の画面名から、テスターが対象画面を一意に特定できるか？
3. 「対象外」の判断理由を第三者が読んで納得できるか？
