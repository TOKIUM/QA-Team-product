# テスト設計パターン

テストケース設計時に参照する6つの基本パターンと設計上の注意点です。

---

## パターン1: CRUD基本フロー

```
前提条件 → 新規作成 → 一覧反映確認 → 詳細確認 → 編集 → 削除 → 一覧から消失確認
```

| バリエーション | 内容 |
|---------------|------|
| 一覧0件 | 空状態から新規作成→反映確認 |
| 一覧1件 | 既存1件+新規作成→反映確認 |
| 一覧複数件 | 複数件+新規作成→一括削除含む |
| 削除操作 | 詳細画面から個別削除 + 一覧から一括削除の両方を含める |

---

## パターン2: 権限マトリクス

```
機能 x ロール の組み合わせ全網羅
```

| 項目 | 内容 |
|------|------|
| 権限種別 | 一般、管理者、集計者、取引先管理者、承認者（複数同時保持可） |
| 制限 | 集計画面→集計者権限必須、システム設定→管理者or集計者、承認画面→承認者権限 |
| 特殊ロール | 代理人（指定メンバーの申請・承認を代理実行） |
| URL直接アクセス | メニュー非表示 ≠ アクセス不可。URL直打ちでの権限チェックも確認 |

---

## パターン3: エラー回復フロー

```
エラー発生 → エラーメッセージ確認 → 再操作 → 正常完了
```

| バリエーション | 内容 |
|---------------|------|
| リトライ可 | エラー→再操作→正常完了を確認 |
| リトライ不可 | エラーメッセージで理由が示されていることが期待値 |

---

## パターン4: 金額計算テスト

```
入力経路別 x 数値パターン x 丸め処理 x 税率設定 の組み合わせ
```

| 軸 | バリエーション |
|----|---------------|
| 入力経路 | 手入力、自動入力(OCR等)、自動仕訳、定期支払、カード連携 |
| 数値 | 整数、小数、負の値、0 |
| 丸め処理 | 切捨て・切上げ・四捨五入（割り切れる/割り切れないケース両方） |
| 按分 | 割合指定→合計が元の金額と一致するか（端数処理に注意） |
| 税率 | 消費税・源泉徴収税の自動計算、税抜/税込切替 |

**重要**: 金額計算テストは必ず **P1** とする。

---

## パターン5: 時間境界テスト

```
境界タイミング直前の操作 → 時間経過 → 境界タイミング直後の状態確認
```

| 境界 | 確認内容 |
|------|----------|
| 日跨ぎ (23:59→0:00) | ステータス変化、日次バッチ |
| 月跨ぎ (月末→翌月1日) | 月次集計、締め処理 |
| 年跨ぎ (12/31→1/1) | 年次処理 |
| 年度跨ぎ (3/31→4/1) | 会計年度処理 |
| 有効期限切れ | セッション、データの期限 |
| 長期間放置 | 機能維持確認 |

---

## パターン6: リリース前後比較テスト

```
リリース前の状態を記録 → リリース実施 → リリース後に同じ操作・データで確認
```

| 確認項目 | 内容 |
|----------|------|
| 既存データ | リリース前の設定・データが参照・操作可能か |
| 設定値 | 意図しない変更（削除、追加）がないか |
| 途中状態 | 申請中・承認待ち等がリリースを跨いで継続処理できるか |
| ロールバック | 元の状態に復帰できるか |

---

## 設計上の注意点

### 事前条件の記載
- テスト実行前の必要なデータ状態を必ず明記
- テスト間のデータ干渉に注意（独立性確保）
- 同じ準備から開始できるケースは並べて記載（効率化）

### 効率的な実行順序
1. 実行に時間がかかるものから
2. 不具合発生時の影響範囲が広いものから
