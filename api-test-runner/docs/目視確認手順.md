# POST系テスト実行後の目視確認手順

API自動テストで `post_normal` や `crud_chain` を有効化した場合に、
TOKIUM画面上でデータが正しく変更・削除されたかを目視確認するための手順。

---

## 前提

- Base URL: `https://dev.keihi.com`（開発環境）
- テナント: 株式会社とみた（APIキー発行済み）
- テスト対象のPOST API:

| API | エンドポイント | 操作内容 |
|-----|---------------|----------|
| 従業員登録 | `members/bulk_create_job` | 従業員を新規登録 |
| 従業員更新 | `members/bulk_update_job` | 既存従業員の情報を更新 |
| プロジェクト登録 | `projects/bulk_create_job` | プロジェクトを新規登録 |
| プロジェクト更新 | `projects/bulk_update_job` | 既存プロジェクトの情報を更新 |

### 自動化された確認機能（v2）

テストツールに以下の自動比較機能が組み込まれている。画面目視は補足確認として行う。

| 機能 | 説明 |
|------|------|
| **リクエストボディ記録** | report.jsonに全テストのrequest_bodyが記録される |
| **識別可能なテストデータ名** | `[APIテスト]{フィールド名}_{日時}` 形式で自動命名 |
| **テスト前後のデータ比較** | APIで自動取得し、件数変化・追加ID・変更フィールドを検出 |
| **バッチジョブポーリング** | ジョブ完了を自動待機（最大30秒）してから比較 |

---

## 1. テスト実行前の事前準備

### 1-1. 画面の現状を記録（任意）

データ比較は自動化されているが、目視確認したい場合は以下を記録。

| # | 画面 | 確認箇所 | 記録内容 |
|---|------|----------|----------|
| 1 | 従業員一覧 | 従業員の総数 | 件数 |
| 2 | プロジェクト一覧 | プロジェクトの総数 | 件数 |

### 1-2. テストデータの設定

`config.yaml` の `post_normal.body_overrides` でAPI別のテスト値を設定済み。
実在するIDを指定することで、実際にデータが変更されるテストになる。

```yaml
post_normal:
  body_overrides:
    members-bulk_update_job:     # 実在する従業員ID
      id: "b3086003-ba46-4ee1-9c25-60cd4d6d69c2"
    members-bulk_create_job:     # 正しいauthorities構造
      authorities:
        is_admin: false
        is_accountant: false
        # ...
    projects-bulk_update_job:    # 実在するプロジェクトUUID
      id: "a9d4556c-9e07-49fd-b3ba-a0b64c043019"
    projects-bulk_create_job:    # 新規登録時はid/type不要
      id: null     # null指定でボディから除外
      type: null
```

## 2. テスト実行

### CLI

```bash
# post_normal のみ有効化して実行（config.yamlのpatternsに追加するか、--patternで指定）
python -m api_test_runner run --pattern post_normal

# crud_chain も有効化する場合
python -m api_test_runner run --pattern post_normal,crud_chain
```

### Web UI

1. `python -m api_test_runner web` でサーバー起動
2. ブラウザで http://localhost:8000 を開く
3. パターン選択で `post_normal` のみ選択
4. 「テスト実行」ボタンをクリック
5. 結果一覧のテスト名をクリック → リクエストボディとデータ比較が表示される

### 自動出力される情報

テスト実行後、コンソールとレポートに以下が出力される:

```
[POLL] ジョブ完了待機中... (job_id: abc123...)
[POLL] ジョブ完了 (6秒)
[PASS] POST /members/bulk_create_job.json normal - expect 200
[DATA] 総件数: 49 → 50 (+1)
[DATA] members(取得分): 49件 → 50件 (追加: 1, 削除: 0, 変更: 0)
```

### 出力ファイル（`results/<timestamp>/` 配下）

| ファイル | 内容 |
|---------|------|
| `report.json` | request_body + data_comparison を含む完全レポート |
| `report.html` | リクエストボディ・データ比較を展開表示可能 |
| `*_before.json` | テスト前のGETスナップショット |
| `*_after.json` | テスト後のGETスナップショット |

## 3. 結果確認

### 3-1. 自動データ比較の確認（推奨）

report.json の `data_comparison` セクションを確認する。

```json
{
  "_total": { "before_total": 49, "after_total": 50, "diff": 1 },
  "members": {
    "before_count": 49, "after_count": 50,
    "added_count": 1, "added_ids": ["新しいUUID"],
    "changed_count": 0, "changed": {}
  }
}
```

| 確認項目 | 見るべき値 |
|---------|-----------|
| データ追加 | `added_count > 0` かつ `added_ids` にIDあり |
| データ更新 | `changed_count > 0` かつ `changed` にフィールド別before/after |
| 件数変化 | `_total.diff` が期待通り |

### 3-2. 画面での目視確認（補足）

自動比較で確認できない点（表示の正確性、UI上の整合性等）を目視で確認。

| API | 画面操作 | 確認ポイント |
|-----|---------|-------------|
| 従業員登録 | 一覧で `[APIテスト]` 検索 | テストデータが存在すること |
| 従業員更新 | 対象IDの詳細画面 | 指定項目が変更、未指定項目が保持 |
| プロジェクト登録 | 一覧で `[APIテスト]` 検索 | テストデータが存在すること |
| プロジェクト更新 | 対象IDの詳細画面 | 指定項目が変更、未指定項目が保持 |

---

## 4. CRUDチェーン（`crud_chain`）の目視確認

crud_chainはPOST→GET→DELETE→GETの4ステップを自動実行する。

| # | 確認タイミング | 操作 | 確認ポイント |
|---|---------------|------|-------------|
| 1 | テスト完了後 | レポートでチェーンの各ステップ結果を確認 | POST(200)→GET(200)→DELETE(200)→GET(404)の順にPASSしていること |
| 2 | テスト完了後 | 画面で登録データを検索 | DELETE済みのため**存在しない**こと（CRUDチェーンは自動クリーンアップ） |
| 3 | DELETE失敗時 | 画面で登録データを検索 | データが**残っている**場合は手動削除が必要 |

---

## 5. テスト後のクリーンアップ

### post_normal で登録されたデータの削除

`post_normal`はデータを登録するだけで削除しない。テスト後に手動でクリーンアップが必要。

| # | 対象 | 手順 |
|---|------|------|
| 1 | テスト従業員 | 従業員一覧 → `[APIテスト]` で検索 → 削除（またはブロック） |
| 2 | テストプロジェクト | プロジェクト一覧 → `[APIテスト]` で検索 → 削除 |

> **ヒント**: テストデータには自動で `[APIテスト]` プレフィックスが付くため、検索・一括削除が容易。

### crud_chain で削除が失敗した場合

レポートの `crud-chain-*` テストで DELETE ステップが FAIL の場合、登録データが残っている。
レポートのレスポンスからIDを取得し、画面から手動削除する。

  - ジョブステータス: completed / failed
