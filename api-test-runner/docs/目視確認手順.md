# POST系テスト実行後の目視確認手順

API自動テストで `post_normal` や `crud_chain` を有効化した場合に、
TOKIUM画面上でデータが正しく変更・削除されたかを目視確認するための手順。

---

## 前提

- Base URL: `https://dev.keihi.com`（開発環境）
- テスト対象のPOST API:

| API | エンドポイント | 操作内容 |
|-----|---------------|----------|
| 従業員登録 | `members/bulk_create_job` | 従業員を新規登録 |
| 従業員更新 | `members/bulk_update_job` | 既存従業員の情報を更新 |
| プロジェクト登録 | `projects/bulk_create_job` | プロジェクトを新規登録 |
| プロジェクト更新 | `projects/bulk_update_job` | 既存プロジェクトの情報を更新 |

---

## 1. テスト実行前の事前準備

### 1-1. 画面の現状をスクリーンショットで記録

テスト実行**前**に以下の画面を開き、データの状態を記録しておく。

| # | 画面 | 確認箇所 | 記録内容 |
|---|------|----------|----------|
| 1 | 従業員一覧 | 従業員の総数、最終ページ | 件数と末尾の従業員名 |
| 2 | プロジェクト一覧 | プロジェクトの総数 | 件数と末尾のプロジェクト名 |
| 3 | バッチジョブ一覧（あれば） | 最新のジョブ状態 | 直近のジョブID・ステータス |

### 1-2. テスト用識別子を決める

テストデータと実データを区別するため、テストデータには識別可能な命名を使う。
例: `[APIテスト] テスト従業員_20260306` / `[APIテスト] テストPJ_20260306`

> **config.yaml の `search.overrides` で値を指定できる。**
> テスト用の値を事前に設定してからテストを実行すること。

---

## 2. テスト実行

```bash
# post_normal のみ有効化して実行
python -m api_test_runner run --pattern post_normal

# crud_chain も有効化する場合（config.yaml で crud_chain.enabled: true にしてから）
python -m api_test_runner run --pattern post_normal,crud_chain
```

テスト完了後、レポート (`results/<timestamp>/report.json`) でPASS/FAILを確認。

---

## 3. 登録テスト（`post_normal`）の目視確認

### 3-1. 従業員登録 (`members/bulk_create_job`)

| # | 操作 | 確認ポイント |
|---|------|-------------|
| 1 | TOKIUM管理画面 → 従業員一覧を開く | - |
| 2 | テスト用の従業員名で検索 | テストで送信した `name` の値で検索 |
| 3 | 検索結果を確認 | テストデータが**登録されている**こと |
| 4 | 従業員の詳細画面を開く | 各項目（メール、従業員番号、部署等）がリクエストボディの値と一致すること |
| 5 | バッチジョブの状態を確認（可能であれば） | ジョブが「完了」ステータスになっていること |

### 3-2. 従業員更新 (`members/bulk_update_job`)

| # | 操作 | 確認ポイント |
|---|------|-------------|
| 1 | 更新対象の従業員IDの詳細画面を開く | テストで送信した `id` の従業員 |
| 2 | 更新された項目を確認 | リクエストボディで指定した項目（name, email等）が**変更後の値**になっていること |
| 3 | 指定しなかった項目を確認 | リクエストで**送信しなかった**項目は元の値のまま変わっていないこと |

### 3-3. プロジェクト登録 (`projects/bulk_create_job`)

| # | 操作 | 確認ポイント |
|---|------|-------------|
| 1 | TOKIUM管理画面 → プロジェクト一覧を開く | - |
| 2 | テスト用のプロジェクト名で検索 | テストで送信した `name` の値で検索 |
| 3 | 検索結果を確認 | テストデータが**登録されている**こと |
| 4 | プロジェクトの詳細画面を開く | `display_id`、割当情報がリクエストボディと一致すること |

### 3-4. プロジェクト更新 (`projects/bulk_update_job`)

| # | 操作 | 確認ポイント |
|---|------|-------------|
| 1 | 更新対象のプロジェクトIDの詳細画面を開く | テストで送信した `id` のプロジェクト |
| 2 | 更新された項目を確認 | name, display_id 等が**変更後の値**になっていること |
| 3 | 指定しなかった項目を確認 | 送信しなかった項目は元の値のまま変わっていないこと |

---

## 4. CRUDチェーン（`crud_chain`）の目視確認

crud_chainはPOST→GET→DELETE→GETの4ステップを自動実行する。

| # | 確認タイミング | 操作 | 確認ポイント |
|---|---------------|------|-------------|
| 1 | テスト完了後 | レポートでチェーンの各ステップ結果を確認 | POST(200)→GET(200)→DELETE(200)→GET(404)の順にPASSしていること |
| 2 | テスト完了後 | 画面で登録データを検索 | DELETE済みのため**存在しない**こと（CRUDチェーンは自動クリーンアップ） |
| 3 | DELETE失敗時 | 画面で登録データを検索 | データが**残っている**場合は手動削除が必要 |

---

## 5. テスト後のクリーンアップ

### post_normal で登録されたデータの削除

`post_normal`はデータを登録するだけで削除しない。テスト後に手動でクリーンアップが必要。

| # | 対象 | 手順 |
|---|------|------|
| 1 | テスト従業員 | 従業員一覧 → テスト用名前で検索 → 削除（またはブロック） |
| 2 | テストプロジェクト | プロジェクト一覧 → テスト用名前で検索 → 削除 |

> **ヒント**: テストデータに `[APIテスト]` プレフィックスを付けておくと検索・一括削除が容易。

### crud_chain で削除が失敗した場合

レポートの `crud-chain-*` テストで DELETE ステップが FAIL の場合、登録データが残っている。
レポートのレスポンスからIDを取得し、画面から手動削除する。
